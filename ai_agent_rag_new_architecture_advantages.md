# 新架构相对一期 Dify Workflow + 传统 RAG 的优势分析

调研日期：2026-06-12

本文用于说明：项目一期采用 Dify Workflow + 传统 RAG 实现智能客服时，为什么下一阶段可以升级为 `pi / Nanobot + 知识图谱 / 语境图谱` 架构，以及新架构能解决哪些一期难以解决的问题。

## 1. 核心结论

新架构的优势不在于“换一个更炫的 Agent 框架”，而在于把一期的 **固定工作流 + 文本相似度检索**，升级成 **可编排智能体 + 结构化知识 + 语境约束 + 可审计决策链**。

一句话概括：

```text
一期：从文档里找相似片段，让大模型总结。
新架构：先理解任务，再按规则、图谱、上下文和工具链完成受控回答。
```

如果一期目标只是把基础文档问答做起来，Dify Workflow + 传统 RAG 是合适的。

但如果下一阶段目标是保险智能客服，尤其涉及保单状态、理赔条件、条款版本、客户身份、材料审核、免责规则、多轮追问、人工转接和合规审计，那么 `pi / Nanobot + 知识图谱 / 语境图谱` 的优势会更明显。

## 2. 总体对比

| 维度 | 一期 Dify Workflow + 传统 RAG | 新架构 pi / Nanobot + 知识图谱 / 语境图谱 |
|---|---|---|
| 核心模式 | 固定工作流 + 文档片段检索 | Agent Loop + Skill 编排 + 图谱检索 + 工具调用 |
| 知识形态 | 文本 Chunk | 文本 Chunk + 实体 + 关系 + 条款 + 证据 + 时间 + 状态 |
| 检索方式 | 向量/BM25 相似度召回 | 向量召回 + 图谱多跳 + 条件过滤 + 权限过滤 |
| 推理能力 | 主要依赖 LLM 总结 | 结构化关系推理 + 语境过滤 + LLM 生成 |
| 适用问题 | 简单 FAQ、说明类问答 | 复杂业务问答、规则问答、多条件判断、流程办理 |
| 多轮能力 | 流程固定，状态表达有限 | 支持会话状态、任务状态、工具结果和长期记忆 |
| 可解释性 | 主要引用文档片段 | 可解释实体路径、证据来源、规则过滤和决策链 |
| 可审计性 | 依赖工作流日志和引用片段 | 可记录输入、检索、图路径、工具调用、规则命中和输出依据 |
| 扩展性 | 节点和分支逐渐膨胀 | 底座、Skill、图谱、工具、审计分层扩展 |
| 主要风险 | 语义相似但业务不适用 | 建模和治理成本更高，需要分阶段落地 |

## 3. 检索能力升级

传统 RAG 最大的问题是：**语义相似不等于业务适用**。

一期流程通常是：

```text
用户问题 -> 文档分块检索 -> 向量/BM25 召回 -> Rerank -> 拼接 Prompt -> LLM 总结
```

这种方式适合简单说明类问题，但对于保险智能客服，用户问题往往不只是“问一段知识”，而是要判断当前场景是否适用某条规则。

例如用户问：

```text
这个产品能不能赔？
```

一期可能召回一段看起来相似的理赔条款，但不一定校验：

1. 产品版本。
2. 保单状态。
3. 事故时间。
4. 等待期。
5. 客户身份。
6. 地区规则。
7. 免责条款。
8. 当前材料是否齐全。

新架构可以把问题拆解为：

```text
客户是谁？
保单是否有效？
产品版本是什么？
事故时间是否在保障期内？
当前条款是否已过期？
是否命中免责规则？
需要哪些材料？
回答依据是哪条证据？
```

这就是语境图谱比普通 RAG 强的地方：它不是只找“相似内容”，而是找“当前语境下可用的知识”。

## 4. 复杂推理能力升级

一期更像单轮问答流水线：

```text
用户问题 -> 检索文档 -> 拼 Prompt -> LLM 回答
```

它不擅长多跳关系，例如：

```text
客户 -> 保单 -> 产品 -> 责任条款 -> 免责条款 -> 理赔材料 -> 当前流程状态
```

知识图谱可以解决“关系链”问题。

语境图谱进一步解决“这条关系链当前是否有效”的问题。

示例：

```text
知识图谱：客户 A 购买了产品 X，产品 X 包含责任条款 C。
语境图谱：客户 A 的保单当前有效，事故发生在等待期后，条款 C 属于 2025 版，当前仍生效，但材料 M 缺失。
```

因此，新架构能从“回答知识”升级为“辅助业务判断”。

## 5. 规则、版本和时效问题

智能客服在保险、制度、法务、合规类场景里最容易出错的地方，不是模型不会总结，而是：

```text
新旧规则混用
不同产品版本混用
不同地区规则混用
不同客户身份混用
历史条款和现行条款混用
```

一期传统 RAG 很容易召回旧文档，只要语义相似就进入 Prompt。

语境图谱可以给知识加上：

```text
生效时间
失效时间
产品版本
适用地区
适用渠道
客户身份
来源文档
审核状态
权限等级
```

这样系统回答前可以先过滤掉“不适用于当前场景”的知识。

这对保险客服尤其关键，因为保险问答通常不是通用知识解释，而是带有强时间、强版本、强身份、强规则约束的业务判断。

## 6. 智能体执行能力升级

Dify Workflow 的优势是配置快、流程清楚，但复杂任务会逐渐变硬：

```text
流程固定
分支膨胀
上下文传递弱
工具调用治理有限
多轮任务状态不够自然
```

pi / Nanobot 这类底座更适合作为 Agent Runtime。

| 能力 | 新架构价值 |
|---|---|
| Agent Loop | 支持多轮规划、工具调用、观察结果、继续执行 |
| Skill 机制 | 把业务 SOP、话术、流程规范做成可复用说明 |
| Tool / MCP 接入 | 可接核心系统、保单查询、理赔进度、规则服务 |
| 上下文治理 | 控制哪些历史、记忆、工具结果进入当前推理 |
| 事件流 / Hook | 便于做审计、拦截、人工确认、异常处理 |

也就是说，新架构不是让模型自由发挥，而是让模型在受控工具、受控流程、受控知识里完成任务。

## 7. 可解释性和审计能力升级

一期通常只能解释：

```text
我引用了哪些文档片段。
```

但业务真正需要的是：

```text
为什么这样判断？
用了哪个版本的条款？
依据来自哪里？
是否经过权限过滤？
是否触发了免责规则？
哪些信息不确定，需要人工确认？
```

语境图谱可以记录：

```text
用户问题 -> 命中实体 -> 相关保单 -> 适用条款 -> 证据片段 -> 规则过滤 -> 最终回答
```

这对线上客服、投诉复盘、合规审计非常重要。

## 8. 幻觉和错误率控制

传统 RAG 降低幻觉的方式主要是“给模型更多上下文”。

但上下文多了也会带来问题：旧规则、冲突文档、无关片段一起塞给模型，反而更容易答错。

新架构的思路是：

```text
先过滤，再生成。
先结构化约束，再让模型表达。
```

在语境图谱下，模型不是随便总结，而是基于当前可用证据回答。

它能显著降低这些错误：

```text
引用过期条款
混用产品版本
忽略免责条件
把相似条款当适用条款
对无权限内容进行回答
没有证据却给确定结论
```

## 9. 用户体验升级

一期客服更像“知识库问答机器人”。

新架构可以更接近“业务助手”：

```text
不仅回答是什么，还能告诉用户下一步怎么做。
不仅解释规则，还能查当前保单 / 工单 / 材料状态。
不仅给结论，还能说明缺什么材料、是否需要人工介入。
```

比如用户问：

```text
我这个情况能不能理赔？
```

一期可能回答一段泛化条款。

新架构可以回答：

```text
根据你当前保单状态、事故时间和产品条款，初步看符合责任范围；
但系统显示缺少医院诊断证明和费用明细，补齐后可进入人工审核。
```

这就是从“知识问答”到“任务办理”的差别。

## 10. 架构扩展性升级

一期 Dify Workflow 做到后期容易出现：

```text
节点越来越多
分支越来越复杂
不同业务线重复搭流程
规则散落在 Prompt 和 Workflow 节点里
问题定位困难
```

新架构可以分层：

```text
pi / Nanobot：Agent Loop、工具调用、上下文管理
Skill：业务 SOP、操作说明、话术规范
知识图谱：稳定事实和实体关系
语境图谱：时间、版本、权限、证据、决策链
MCP / 工具层：保单、理赔、客户、工单等业务系统
审计层：记录输入、检索、工具调用、回答依据
```

这样每层职责更清楚，后续扩展产品线、接新系统、加新规则会更稳。

## 11. 能解决的一期核心问题

| 一期问题 | 新架构解决方式 |
|---|---|
| 文档片段碎片化 | 知识图谱建立实体和关系 |
| 多跳问题答不好 | 图谱路径查询和关系推理 |
| 新旧条款混用 | 语境图谱加版本、生效期、失效期 |
| 相似但不适用 | 按产品、身份、地区、渠道、任务状态过滤 |
| 回答不可审计 | 保存证据、来源、规则、决策链 |
| 工作流复杂后难维护 | Agent + Skill + 工具分层编排 |
| 无法办理复杂任务 | Agent 可多步调用工具、查询状态、推进流程 |
| 幻觉偏高 | 先结构化约束，再基于证据生成 |
| 人工复核成本高 | 输出依据、不确定点和人工介入条件 |
| 业务能力难复用 | Skill、图谱、工具可跨业务线复用 |

## 12. 推荐演进路线

最推荐的演进方式不是推倒重来，而是分层升级：

```text
保留一期标准 RAG 作为文本召回基础
        ↓
引入知识图谱解决实体关系和多跳问题
        ↓
引入语境图谱解决时间、版本、权限、证据和决策链
        ↓
用 pi / Nanobot 承载 Agent Loop、Skill、工具调用和审计治理
```

对应落地步骤可以分为四个阶段：

| 阶段 | 目标 | 关键建设内容 |
|---|---|---|
| 阶段 1 | 保留一期能力，补齐评测和日志 | 继续使用 Dify RAG，建立问题集、错误类型、召回日志和人工标注 |
| 阶段 2 | 引入知识图谱 | 抽取产品、条款、保单、材料、免责规则等实体关系，支持多跳查询 |
| 阶段 3 | 引入语境图谱 | 增加版本、生效期、失效期、来源、权限、客户身份、任务状态 |
| 阶段 4 | 引入智能体底座 | 用 pi / Nanobot 接管复杂任务编排、工具调用、Skill、审计和人工确认 |

## 13. 最终判断

如果只做简单知识库问答，一期 Dify Workflow + 传统 RAG 足够。

如果要做真正可用于保险业务的智能客服，新架构更有价值：

```text
Dify Workflow + 传统 RAG 解决“能不能答”
pi / Nanobot + 知识图谱 / 语境图谱解决“能不能答对、能不能解释、能不能办理、能不能审计”
```

因此，新架构不是对一期的否定，而是对一期的增强：

1. 标准 RAG 继续作为文本证据召回层。
2. 知识图谱补齐实体关系和多跳推理。
3. 语境图谱补齐时间、版本、权限、证据和决策链。
4. pi / Nanobot 补齐复杂任务编排、工具调用、Skill 复用和审计治理。

最终目标是把智能客服从“文档问答系统”升级为“可控、可解释、可审计、可办理业务的智能体系统”。
