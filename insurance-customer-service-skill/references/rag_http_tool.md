# HTTP RAG 工具契约

## 目的

本参考文档定义保险智能客服技能如何通过 HTTP 调用 RAG 检索工具。所有通用问答、产品条款解释和产品对比都必须遵循此契约。

## 工具定位

RAG 是外部检索工具，负责从保险知识库返回可引用证据。模型负责：

- 构造检索请求。
- 判断证据是否支持回答。
- 基于证据生成自然语言回答。
- 在证据不足或工具失败时安全降级。

模型不得把自身记忆当作 RAG 证据。

## 推荐配置

由运行环境或宿主系统提供以下配置：

```text
RAG_HTTP_ENDPOINT=https://example.internal/rag/search
RAG_HTTP_API_KEY=由宿主系统注入
RAG_HTTP_TIMEOUT_MS=5000
RAG_HTTP_TOP_K=5
```

如果实际系统使用其他配置名，应在接入层映射到同等含义。

## HTTP 请求

使用 `POST` 请求。

推荐请求头：

```http
Content-Type: application/json
Authorization: Bearer ${RAG_HTTP_API_KEY}
X-Request-Source: insurance-customer-service-skill
```

推荐请求体：

```json
{
  "query": "用户问题或经过必要改写后的检索问题",
  "intent": "general_qa | product_terms | product_compare | safety_policy",
  "top_k": 5,
  "filters": {
    "product_names": [],
    "clause_names": [],
    "insurance_types": [],
    "business_tags": [],
    "jurisdiction": null
  },
  "context": {
    "conversation_summary": "只包含必要的非敏感上下文",
    "comparison_dimensions": [],
    "retrieval_round": 1,
    "missing_information": [],
    "original_question": "用户原始问题",
    "language": "zh-CN"
  }
}
```

## 请求构造规则

- `query` 应聚焦检索目标，不要把整段无关对话原样发送。
- `intent` 必须与路由结果一致。
- `filters.product_names` 只放用户明确提到的产品名，不要猜测产品。
- `filters.clause_names` 只放用户明确提到或上下文明确的条款/责任。
- `context.conversation_summary` 只保留完成检索所需的非敏感信息。
- `context.retrieval_round` 标明当前是第几轮检索。
- `context.missing_information` 在第二轮和第三轮中说明上一轮证据缺口。
- `context.original_question` 用于保持多轮检索不偏离用户原始问题，不得包含敏感个人信息。
- 不要发送客户身份证号、手机号、保单号、银行卡号、住址、病历号、案件号等敏感个人信息。

## HTTP 响应

推荐响应体：

```json
{
  "request_id": "rag-20260628-0001",
  "status": "ok",
  "evidence": [
    {
      "id": "doc-chunk-001",
      "title": "知识库文档标题",
      "content": "可用于回答的片段内容",
      "source": "文档来源或知识库名称",
      "score": 0.86,
      "metadata": {
        "product_name": "产品名称",
        "clause_name": "条款或责任名称",
        "doc_type": "faq | clause | guide | policy",
        "url": null
      }
    }
  ],
  "warnings": [],
  "error": null
}
```

## 证据可用性判定

只有满足以下条件的内容才能作为回答依据：

- `status` 为 `ok`。
- `evidence` 非空。
- 证据片段与用户问题直接相关。
- 证据内容足以支持回答中的关键结论。
- 产品条款、责任、免责、费用、等待期、理赔条件等关键结论必须有直接证据。

以下情况视为证据不足：

- 只有标题命中，正文没有支持。
- 只有相似产品证据，没有当前产品证据。
- 证据只解释通用概念，但用户问的是具体条款责任。
- 证据分数高但内容与问题不直接相关。
- 响应字段缺失、格式错误或无法解析。

## 多次检索策略

通用检索场景必要时最多进行三轮检索：

1. 第一轮使用用户原始问题和路由意图检索。
2. 如果证据不足，先判断“还缺什么信息才能回答原始问题”，再构建第二轮检索问题。
3. 如果第二轮后仍不足，基于前两轮证据缺口构建第三轮检索问题。

第三轮仍然无足够证据时，必须按证据不足处理。不要无限重试。

产品对比场景可以分别检索每个产品，但每个产品最多两轮检索。某个产品缺少证据时，应在对比表中标注“当前知识库未找到”，不要用另一个产品或相似产品补齐。

### 下一轮检索构建规则

下一轮检索必须围绕证据缺口构建，而不是简单重复上一轮请求。

构建方法：

- 将复合问题拆成多个关键子问题。
- 将口语表达改写成保险知识库常用术语。
- 补充同义词或业务近义词。
- 针对缺失维度检索，例如定义、适用条件、例外、材料、流程、限制。
- 保留用户原始问题中的险种、业务场景和限定条件。

示例：

```text
原始问题：重疾险和医疗险有什么区别，理赔时分别看什么？

第 1 轮 query：重疾险和医疗险有什么区别
如果只检索到定义，缺少理赔维度：
第 2 轮 query：重疾险 医疗险 理赔条件 理赔材料 区别
如果仍缺少赔付方式：
第 3 轮 query：重疾险 给付型 医疗险 报销型 赔付方式 区别
```

### 多轮证据综合规则

最终回答必须回到用户原始问题，并综合所有轮次的有效证据：

- 去重相同或重复证据。
- 合并不同轮次中互补的证据。
- 标出仍未被证据支持的部分。
- 如果不同轮次证据冲突，按证据冲突处理，不自行选择结论。
- 不要只依据最后一轮检索结果回答。

## 超时、重试和错误处理

推荐生产参数：

- 单次超时：5 秒。
- 最多重试：1 次。
- 只对网络抖动、5xx、429 等可恢复错误重试。
- 不对 400、401、403 等请求或权限错误盲目重试。

错误处理：

- HTTP 超时：说明当前知识库检索暂时不可用。
- 401/403：说明当前检索服务未授权或不可用，不暴露密钥细节。
- 4xx 请求错误：检查请求构造；面向用户时只说明暂时无法检索。
- 5xx 服务错误：说明知识库服务暂时异常。
- 响应格式错误：视为工具失败，不使用其中内容。

## 安全与隐私

- 不向 RAG 工具发送敏感个人信息。
- 不在最终回答中暴露 API 地址、密钥、内部请求头或完整原始响应。
- 可以引用文档标题、来源名称或片段摘要，但不要展示内部调试字段。
- 如果用户问题包含敏感个人信息，应先按主技能的客户个人问题边界处理，不要直接转发给 RAG 工具。

## 最终回答引用方式

最终回答不要求机械展示每条证据，但必须能追溯到 `evidence`。

建议表达：

```text
根据当前知识库检索到的资料，...
```

当证据不足：

```text
当前知识库没有检索到足够资料确认这一点，因此不能给出确定结论。
```

当工具失败：

```text
当前知识库检索暂时不可用，我无法基于可靠证据回答这个问题。建议稍后重试，或通过官方渠道确认。
```

## 调用记录建议

生产环境建议记录以下非敏感字段，便于排查和评测：

- 请求时间。
- 路由意图。
- 检索轮次。
- 脱敏后的查询文本。
- RAG `request_id`。
- HTTP 状态码。
- 命中文档数量。
- 是否进入证据不足、工具失败或安全降级。

不要记录 API 密钥、敏感个人信息或完整客户隐私文本。
