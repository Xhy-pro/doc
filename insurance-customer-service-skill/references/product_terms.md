# 产品条款问答子技能

## 目的

本参考文档用于处理关于产品条款、保障责任、免责条款、定义、等待期、给付规则和理赔条件的通用问题。

## 适用场景

当用户询问以下内容时，使用本子技能：

- 这个产品保什么？
- 哪些情况不保？
- 等待期是多久？
- 某项责任的理赔条件是什么？
- 某个条款术语是什么意思？
- 产品如何定义疾病、意外、住院、伤残或其他保险事故？

## 不适用场景

不要使用本子技能处理：

- 客户个人理赔结果。
- 个人保单状态或已购买计划详情。
- 个性化购买建议。
- 产品之间的对比。此类问题使用 `product_compare.md`。
- 不安全或违法请求。此类问题使用 `safety_boundary.md`。

## 输入要求

回答前尽量识别：

- 产品名称。
- 条款名称或责任名称。
- 用户的具体问题。
- 检索到的条款证据。
- 证据是否充分。

如果缺少产品名称或条款对象，且答案依赖这些信息，只问一个澄清问题。

## 处理流程

1. 确认问题是关于公开/通用产品条款，而不是个人保单。
2. 按 `rag_http_tool.md` 调用 HTTP RAG 工具检索相关产品条款证据，`intent` 使用 `product_terms`。
3. 只提取回答该问题所需的条款要点。
4. 用通俗语言解释条款含义。
5. 保留重要条件、免责、限制和依赖关系。
6. 不要超出检索条款扩展保障范围。
7. 如果证据缺失或冲突，不要推断产品责任。

## RAG 要求

回答必须基于检索到的产品资料。对于产品责任、免责、等待期、理赔条件、给付金额和限制类问题：

- 不要根据通用保险常识作答。
- 不要合并相似产品中未被证据支持的内容。
- 除非证据直接支持，否则不要声称某项责任存在。
- 当免责或条件与用户问题相关时，不要省略。
- 如果 HTTP RAG 工具失败、超时或返回格式异常，视为无法取得条款证据。

## HTTP RAG 调用要点

- `filters.product_names` 填入用户明确提供的产品名称。
- `filters.clause_names` 填入用户明确提到的责任、条款或概念。
- `filters.business_tags` 可填入 `coverage`、`exclusion`、`waiting_period`、`claim_condition` 等标签。
- 当产品名称缺失且答案依赖具体产品时，先澄清，不要直接调用宽泛检索后猜测。
- 不要把客户个人病史、保单号、案件号等敏感信息发送给 RAG 工具。

## 回答模式

推荐结构：

```text
根据当前检索到的资料，这个条款/责任可以这样理解：...

需要注意的是：...

如果你问的是具体某一份已购买保单，实际结果还需要以该保单条款和官方查询为准。
```

使用自然语言表达，标题可选。

## 澄清触发条件

出现以下情况时，只问一个问题：

- 多个产品可能匹配用户的说法。
- 用户说“这个产品”，但没有提供产品名称。
- 用户问“保不保”，但没有说明具体责任或场景。
- 当前证据对应的条款与用户意图可能不一致。

## 拒答或降级触发条件

出现以下情况时，降级为通用解释：

- 用户问“我这种情况能不能赔？”
- 用户问“我的保单会不会赔？”
- 用户问“我能不能隐瞒这个情况？”
- 用户要求法律或医疗结论。

## 示例

用户：“这个重疾险保不保原位癌？”

安全处理：

```text
需要先确认具体产品名称或条款名称。不同重疾险对原位癌、轻症、恶性肿瘤相关责任的定义可能不同。请提供产品名称，我可以基于知识库里的条款资料帮你做通用解释。
```

用户：“等待期内出险一定不赔吗？”

安全处理：

```text
不能简单说一定不赔。根据具体产品条款，等待期内发生不同类型的保险事故，处理方式可能不同，例如解除合同、退还保费、部分责任不承担等。需要结合具体产品条款判断。
```

## 输出质量检查清单

- 产品或条款对象清楚，或者已经提出澄清问题。
- 回答基于检索到的条款证据。
- 没有隐藏条件和免责。
- HTTP RAG 工具失败时没有编造条款内容。
- 没有给出客户个人理赔结论。
- 没有给出个性化购买建议。
