# 知识图谱与语境图谱对比

调研日期：2026-06-11

本文从技术选型角度对比知识图谱和语境图谱，说明二者的相同点、区别、语境图谱主要解决知识图谱的哪些不足，并通过具体例子说明二者在业务语义和工程落地上的差异。

## 1. 核心理解

可以先用一句话区分二者：

```text
知识图谱 = 组织“事实和关系”
语境图谱 = 组织“事实在什么情境下成立、如何被使用、为什么得出结论”
```

知识图谱更像企业或领域的“事实知识库”。它关注实体、概念和实体之间的关系。

语境图谱更像“上下文和决策记忆系统”。它不只关注事实本身，还关注事实的时间、来源、证据、任务、权限、状态、用户意图和决策过程。

二者不是简单替代关系。更合理的理解是：

```text
知识图谱是稳定事实层
语境图谱是建立在事实层之上的上下文层、时间层、证据层和决策层
```

## 2. 相同点

| 维度 | 相同点 |
|---|---|
| 数据结构 | 都使用图结构表达信息，包含节点、边和属性 |
| 核心对象 | 都可以表达实体、概念、事件和关系 |
| 核心能力 | 都支持路径查询、多跳查询、关系推理和语义检索 |
| 技术底座 | 都可以使用 Neo4j、JanusGraph、TigerGraph、RDF、GraphDB、Stardog 等实现 |
| 查询语言 | 都可能使用 Cypher、Gremlin、SPARQL 或自定义查询接口 |
| 应用方向 | 都可用于搜索、推荐、问答、风控、RAG、数据治理和企业知识管理 |
| 与大模型结合 | 都可以为 LLM 提供结构化信息，弥补纯向量检索的不足 |
| 数据治理 | 都需要关注实体统一、关系建模、数据质量和权限控制 |

## 3. 核心区别

| 维度 | 知识图谱 | 语境图谱 |
|---|---|---|
| 核心问题 | 有什么事实 | 在什么语境下，这个事实或决策成立 |
| 关注重点 | 实体、概念、属性、关系 | 实体、关系、事件、任务、证据、状态、决策链 |
| 典型节点 | 人、公司、商品、疾病、文档、地点 | 用户、任务、会话、事件、证据、工具调用、决策、状态 |
| 典型边 | 属于、购买、投资、治疗、引用、任职于 | 触发、支持、反驳、来源于、发生于、有效于、依赖于 |
| 时间维度 | 可以有时间，但通常不是核心 | 时间、版本、有效期通常是核心 |
| 来源追踪 | 可以做，但不是所有知识图谱都强调 | 通常是核心能力 |
| 数据稳定性 | 相对稳定 | 更动态，和任务、时间、会话、状态强相关 |
| 查询目标 | 查事实、查路径、做语义推理 | 查上下文、查原因、查证据、查状态、查先例 |
| 决策解释 | 能解释实体关系路径 | 能解释当前场景下为什么这样判断 |
| 用户上下文 | 通常不是重点 | 是重点 |
| 任务上下文 | 通常不是重点 | 是重点 |
| 典型应用 | 企业知识库、实体搜索、推荐、股权穿透、风控 | Agent 记忆、GraphRAG、审计追踪、流程自动化、上下文工程 |
| 工程位置 | 领域知识层 | 运行时上下文层和决策记忆层 |

## 4. 二者关系

知识图谱和语境图谱可以分层理解：

```text
业务数据 / 文档 / 日志 / 会话
        ↓
实体抽取、关系抽取、实体对齐
        ↓
知识图谱：稳定实体、概念、事实、关系
        ↓
语境图谱：时间、来源、证据、任务、权限、状态、决策链
        ↓
搜索 / 问答 / RAG / Agent / 审计 / 推荐 / 风控
```

如果没有知识图谱，语境图谱容易缺少稳定的实体和关系基础。

如果只有知识图谱，系统可能知道“事实是什么”，但不知道“这个事实在当前任务下是否适用”。

## 5. 语境图谱解决知识图谱的什么问题

### 5.1 解决事实缺少时间语境的问题

知识图谱可能只记录：

```text
公司A --投资--> 公司B
```

但现实业务里更需要知道：

```text
公司A --投资--> 公司B
投资时间：2021-03-10
持股比例：20%
数据来源：工商系统
最后更新时间：2024-12-01
当前状态：已退出
```

如果只看知识图谱，系统可能误判公司 A 现在仍然持有公司 B。

语境图谱会进一步表达：

```text
投资关系R --发生于--> 2021-03-10
投资关系R --来源于--> 工商系统
投资关系R --当前状态--> 已退出
投资关系R --失效时间--> 2024-12-01
```

因此，语境图谱可以告诉系统：这个关系曾经成立，但当前已经失效。

### 5.2 解决事实缺少来源和证据的问题

知识图谱记录：

```text
产品X --适用于--> 行业Y
```

语境图谱会进一步记录：

```text
该结论 --来源于--> 文档D
文档D --版本--> v3.2
证据片段 --位于--> 第12页
文档D --更新时间--> 2026-05-01
该结论 --审核人--> 专家A
```

这样在问答、审计或合规场景中，系统不只是回答“是”，还能回答：

1. 为什么是。
2. 依据在哪里。
3. 来源是否可信。
4. 文档是否过期。
5. 是否经过人工审核。

### 5.3 解决不同场景下结论不同的问题

知识图谱里可能有：

```text
用户A --偏好--> 简洁回答
```

但语境图谱会表达得更细：

```text
用户A --偏好--> 简洁回答
适用场景：代码调试、技术选型
不适用场景：正式报告、论文写作
来源：历史对话
最后确认时间：2026-06-11
```

同一个用户，在不同任务下可能需要不同风格。

知识图谱表达“偏好事实”，语境图谱表达“偏好在什么上下文中成立”。

### 5.4 解决 RAG 只知道相似、不知道关系路径的问题

普通 RAG 可能通过向量相似度召回一段文档，但它不一定知道这段文档和当前问题的业务关系。

知识图谱可能知道：

```text
政策A --涉及--> 产品X
产品X --属于--> 部门B
```

语境图谱可以把当前问题也纳入图中：

```text
用户问题Q --命中--> 产品X
产品X --关联--> 政策A
政策A --证据片段--> Chunk-12
Chunk-12 --来自--> 文档v2
文档v2 --权限级别--> internal
回答R --基于--> Chunk-12
回答R --生成于--> 会话S
```

这样系统不只是“找到相关文档”，还知道：

1. 当前回答用了哪些证据。
2. 证据来自哪个文档版本。
3. 当前用户是否有权限。
4. 证据和问题之间的关系路径是什么。
5. 回答是在什么会话和任务中生成的。

### 5.5 解决决策不可追溯的问题

知识图谱可以表达：

```text
客户A --关联--> 风险事件E
```

语境图谱可以表达：

```text
风险判断D --输入--> 客户A
风险判断D --命中规则--> 规则R1
规则R1 --依赖--> 风险事件E
风险事件E --来源于--> 法院公告
风险判断D --生成时间--> 2026-06-11
风险判断D --审核人--> 审核员B
风险判断D --结论--> 中风险
```

这样可以回答“为什么判为中风险”，而不只是回答“客户有关联风险”。

## 6. 举例说明区别

### 6.1 企业风控

知识图谱回答：

```text
客户A -> 任职于 -> 公司B -> 投资 -> 公司C -> 涉诉
```

它能说明客户 A 和风险公司 C 有三跳关系。

语境图谱进一步回答：

```text
这条路径来自工商数据源
公司B 投资公司C 的关系发生在 2020 年
该投资关系已于 2024 年退出
涉诉信息来自法院公告
公告更新时间是 2026-03-01
当前风控规则要求只看两年内有效关系
因此该风险路径仅作为历史参考，不作为当前高风险判定依据
```

区别：

| 图谱类型 | 回答重点 |
|---|---|
| 知识图谱 | 客户 A 和风险公司 C 有关联 |
| 语境图谱 | 这个关联当前是否有效、能否用于风险判定 |

### 6.2 智能客服

知识图谱：

```text
订单123 --购买了--> 商品X
商品X --支持--> 7天无理由退货
```

语境图谱：

```text
用户当前问题：想退货
订单123 签收时间：2026-06-01
当前时间：2026-06-11
商品X 退货政策：7天无理由
用户已超过 7 天
但该用户是 VIP
客服规则允许 VIP 额外延长 3 天
当前仍可发起人工审核
```

区别：

| 图谱类型 | 回答重点 |
|---|---|
| 知识图谱 | 商品 X 支持 7 天无理由退货 |
| 语境图谱 | 当前订单、当前时间、用户身份和客服规则综合后应该怎么处理 |

### 6.3 Agent 长期记忆

知识图谱：

```text
用户A --使用--> Markdown
用户A --关注--> 技术选型
```

语境图谱：

```text
用户A 在当前项目中连续要求输出 Markdown 文档
当前工作目录是 /Users/henry/Documents/GitHub/doc
已生成 knowledge_graph_technical_selection.md
已生成 context_graph_technical_selection.md
当前任务是在比较知识图谱和语境图谱
用户偏好：中文、结构化、方便技术选型
```

区别：

| 图谱类型 | 回答重点 |
|---|---|
| 知识图谱 | 用户有 Markdown 和技术选型相关偏好 |
| 语境图谱 | 当前线程、当前项目、已完成产物、用户连续意图和下一步任务状态 |

### 6.4 企业文档问答

知识图谱：

```text
制度文档A --适用于--> 部门B
制度文档A --包含--> 条款C
```

语境图谱：

```text
用户问题Q --询问--> 报销额度
用户Q所属部门 --等于--> 部门B
制度文档A --版本--> v4.1
制度文档A --生效时间--> 2026-01-01
条款C --支持--> 回答R
回答R --引用--> 条款C
回答R --生成时间--> 2026-06-11
回答R --权限检查--> 通过
```

区别：

| 图谱类型 | 回答重点 |
|---|---|
| 知识图谱 | 文档 A 包含条款 C，适用于部门 B |
| 语境图谱 | 当前用户是否适用该制度、引用的是哪个版本、回答基于哪条证据 |

### 6.5 IT 运维

知识图谱：

```text
服务A --依赖--> 服务B
服务B --依赖--> 数据库C
```

语境图谱：

```text
告警E --发生于--> 2026-06-11 10:30
告警E --影响--> 服务A
服务A --当前版本--> v2.8
服务B --最近变更--> 2026-06-11 10:10
变更单CHG-001 --修改--> 服务B配置
数据库C --指标--> 连接数正常
历史案例H --相似于--> 当前告警E
历史案例H --解决方案--> 回滚服务B配置
```

区别：

| 图谱类型 | 回答重点 |
|---|---|
| 知识图谱 | 服务之间的依赖关系 |
| 语境图谱 | 当前告警、近期变更、历史案例和可能处理动作 |

## 7. 技术选型角度的判断

| 场景 | 更适合 | 原因 |
|---|---|---|
| 企业知识沉淀 | 知识图谱 | 重点是稳定实体、概念和关系 |
| 实体搜索 | 知识图谱 | 重点是实体识别、消歧和关系查询 |
| 股权穿透 | 知识图谱 | 重点是多跳关系和路径查询 |
| 推荐系统 | 知识图谱 | 重点是用户、商品、内容之间的关系 |
| 文档问答需要证据链 | 语境图谱 | 重点是来源、版本、证据和当前问题上下文 |
| Agent 长期记忆 | 语境图谱 | 重点是会话、用户偏好、任务状态和记忆有效范围 |
| 决策审计 | 语境图谱 | 重点是输入、规则、证据、审批和结论 |
| 流程自动化 | 语境图谱 | 重点是当前状态、历史动作、下一步建议 |
| 数据血缘 | 语境图谱 | 重点是来源、处理过程、影响范围 |
| 普通报表和聚合 | 都不一定需要 | 数仓、OLAP、关系库可能更合适 |

## 8. 典型组合架构

在企业落地中，二者经常组合使用。

```text
结构化数据 / 文档 / 日志 / 会话
        ↓
实体抽取、关系抽取、事件抽取
        ↓
知识图谱：实体、概念、稳定关系
        ↓
语境图谱：时间、来源、证据、任务、状态、权限、决策链
        ↓
应用：搜索、问答、RAG、Agent、风控、审计、推荐
```

可以进一步拆成两层：

| 层级 | 主要内容 | 典型技术 |
|---|---|---|
| 知识图谱层 | 实体、概念、关系、本体、稳定事实 | RDF、Neo4j、GraphDB、Stardog、JanusGraph |
| 语境图谱层 | 会话、任务、事件、证据、时间、权限、决策链 | Neo4j、GraphRAG、Graphiti、LangGraph、OpenTelemetry、PROV-O |

## 9. 快速结论

知识图谱主要解决：

```text
实体是什么？
实体之间有什么关系？
有哪些可复用的领域事实？
```

语境图谱主要解决：

```text
这个事实什么时候成立？
来源是什么？
当前任务是否适用？
当前用户是否有权限使用？
这个回答为什么这么生成？
过去类似场景怎么处理？
```

最终建议：

1. 如果目标是沉淀领域事实、做实体关系查询，优先建设知识图谱。
2. 如果目标是提升 RAG、Agent、审计、流程自动化的上下文理解能力，需要语境图谱。
3. 如果已有知识图谱，可以在其上增加时间、来源、证据、任务、状态、权限和决策链，逐步演进为语境图谱。
4. 如果只是普通 FAQ 或简单文档检索，不建议一开始就上语境图谱。
5. 最实用的企业方案通常是：知识图谱做事实层，语境图谱做上下文层。
