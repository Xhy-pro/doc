## 一、智能体底座能力对比（pi-mono / Nanobot）& Skill 运行机制

### 1.1 本节结论与选型口径

本节基于当前目录内的 pi-mono、Nanobot、Skill 生命周期、上下文压缩与保险客服架构材料整理。核心判断如下：

| 结论项 | 判断 | 对项目选型的含义 |
|---|---|---|
| 底座定位 | **pi-mono 更像 TypeScript Agent Runtime + Coding Agent 产品壳；Nanobot 更像 Python 轻量 Agent 工作台 + MCP/上下文治理底座。** | 二者不是“谁完全替代谁”的关系，而是底层技术栈、产品形态和上下文治理方式不同。 |
| Skill 本质 | **两者的 Skill 都不是直接执行的代码插件，而是给 LLM 阅读的 Markdown 操作说明书。** | 真正可执行的是工具、MCP 服务、扩展或业务 API；Skill 负责告诉模型何时、如何调用这些能力。 |
| Skill 加载 | **两者都采用渐进式加载：索引常驻，正文按需读取。Nanobot 额外支持 `always` Skill 全文常驻。** | 对多数业务 Skill，应把 `description` 写清触发条件，并让模型需要时重读。 |
| 可控性 | **仅靠 pi-mono 或 Nanobot 原生能力，都不足以直接满足保险级流程、权限、审计和合规要求。** | 必须在企业层新增策略层、身份委托、MCP 工具网关、审计事件、人工确认和任务状态模型。 |
| 推荐方向 | **项目主方案宜采用“龙虾/OpenClaw 类 Skill 驱动机制”：底座负责 Agent Loop、工具调用和上下文治理，企业层负责状态机、策略、SOP 和审计。** | Nanobot 更适合作为轻量可嵌入的 Python Agent Loop 与 MCP 接入底座；pi-mono 可作为 TypeScript Runtime、事件流和工具治理参考。 |

### 1.2 pi-mono、Nanobot、百度千帆简易对比

百度千帆在本节中只作为行业云上通用智能体参照，不与 pi-mono、Nanobot 处于完全同一层级。pi-mono 和 Nanobot 都可以私有化改造；千帆更偏托管式平台能力。

| 对比项        | pi-mono                                                                                                                           | Nanobot                                                                                                         | 百度千帆智能体底座                              |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- | -------------------------------------- |
| 核心定位       | Pi Agent Harness Mono Repo。底层由 `pi-ai`、`pi-agent-core`、`pi-coding-agent` 分层组成，重点是模型抽象、Agent Runtime、事件流、会话和 Coding Agent 产品化。     | 轻量 Python Agent 工作台。重点是多渠道接入、会话化 AgentLoop、ContextBuilder、AgentRunner、工具注册、MCP、上下文压缩和 Skill 系统。                 | 云上托管通用智能体平台。重点是低门槛配置、模型自主规划、工具调用和快速上线。 |
| 技术栈与部署     | TypeScript / Node.js，MIT 开源，可自托管。适合已有前端、Node、CLI/RPC 生态团队。                                                                        | Python，MIT 开源，可自托管。适合接入数据工程、图处理、MCP、评测和企业后端服务。                                                                  | 云平台托管，私有化和底层可控性取决于平台能力。                |
| Agent Loop | `pi-agent-core` 提供有状态、事件驱动、支持 tool calling 的 runtime；可通过 `beforeToolCall` / `afterToolCall` 接入治理。                                 | `AgentLoop` 负责会话调度，`AgentRunner` 负责模型 + 工具循环，每次请求前做上下文治理。                                                       | 主要由平台和模型承担自主规划，底层 loop 细节通常不可见。        |
| 工具体系       | `AgentTool` 具备结构化 schema、执行函数、串行/并行模式和事件流；`pi-coding-agent` 预置 read/write/edit/bash/grep/find/ls 等开发工具。                           | ToolRegistry 统一管理内置工具和 MCP 工具，支持 MCP stdio/HTTP、工具过滤、超时和运行时上下文。                                                 | 平台工具配置为主，适合轻量集成，但精细治理能力依赖平台暴露范围。       |
| Skill 机制   | Skill 主要在 `pi-coding-agent` 资源体系中实现。默认以 `<available_skills>` 索引进入 system prompt，模型用 `read` 按需读取；显式 `/skill:name` 可把正文直接展开进当前用户消息。 | Skill 是一组 `SKILL.md` 指南。`always` Skill 全文进入 `# Active Skills`；普通 Skill 只进入 `# Skills` 索引，模型用 `read_file` 按需读取。  | 更接近工具/组件配置，不是本地文件式 Skill 生命周期机制。       |
| 会话与上下文     | `pi-coding-agent` 有 `.jsonl` session、树状 entry、fork/resume/compact；`pi-agent-core` 本身不负责持久化。                                       | session 保存完整历史，但每轮只 replay 合法窗口；Runner 做 microcompact、tool result budget、snip；后台 consolidation 和 memory 负责长期整理。 | 会话和记忆由平台能力决定，企业对底层裁剪、压缩和持久化策略可控性较弱。    |
| 可观测性       | `agent_start`、`turn_start`、`message_*`、`tool_execution_*` 等事件流清晰，适合 UI、日志和审计映射。                                                   | Agent Hook、流式输出、工具执行阶段可观察；事件可接入企业审计，但需要企业层标准化。                                                                  | 依赖平台日志与监控能力。                           |
| 原生边界       | 不内置保险知识、权限隔离、租户治理、业务审计；默认 Coding Agent 工具不应直接进入保险生产。                                                                              | 不内置保险任务图、企业运行图、决策语境图谱、保险级授权和审计；默认个人助理、文件、Shell、网页工具不应直接进生产工具目录。                                                 | 底层可控性、私有化、审计和数据治理边界受云平台限制。             |
| 适配场景       | TypeScript 技术栈、需要强事件流、强产品会话、Coding Agent 或开发者工具类场景。                                                                               | Python 技术栈、MCP 接入、长上下文治理、多模型路由、企业后端和图/数据服务集成场景。                                                                 | 快速演示、轻量通用问答、无严格 SOP 的云上场景。             |

**小结**：如果目标是保险客服、制度问答、理赔咨询等强约束场景，不能采用“模型自由规划 + 工具自由调用”的黑盒方式。更合理的路线是：以 Nanobot 或 pi-mono 作为 Agent Loop 与工具调用底座，在其上建设 OpenClaw 类状态机、策略层、MCP 工具网关和可审计 Skill 编排。

### 1.3 两款开源底座的能力分层

#### 1.3.1 pi-mono 能力分层

pi-mono 不是单一应用，而是一套围绕 Agent 构建的 monorepo。关键分层如下：

| 层级 | 主要模块 | 原生能力 | 对项目的价值 |
|---|---|---|---|
| 模型与协议层 | `pi-ai` | 多 provider 统一调用、流式响应、工具协议、模型元数据、usage/cost/thinking 等事件抽象。 | 便于在私有模型、OpenAI-compatible 网关和公开模型之间做适配与切换。 |
| Agent Runtime 层 | `pi-agent-core` | 有状态 `Agent`、`AgentState`、`AgentMessage`、`AgentTool`、tool loop、事件流、串行/并行工具执行、`beforeToolCall` / `afterToolCall`。 | 可作为企业 Agent 内核参考，尤其适合做工具前置拦截、结果清洗和审计事件映射。 |
| 产品编排层 | `pi-coding-agent` | CLI/TUI/RPC/SDK、SessionManager、AgentSession、资源加载、Skills、Extensions、Prompts、Themes、Settings、Auth。 | 展示如何把 runtime 组织成长期可恢复的产品，但默认定位是 Coding Agent。 |
| 企业业务层 | 需新增 | 保险知识图谱、MCP 工具网关、业务策略、身份授权、客户确认、人工接管、审计和数据治理。 | pi 原生不提供这层，必须由项目自建。 |

pi-mono 的强项是 Runtime 抽象清晰、事件流完整、工具钩子明确、产品会话能力强；局限是业务治理、权限隔离、保险知识和生产工具目录都需要企业层补齐。

#### 1.3.2 Nanobot 能力分层

Nanobot 可以理解为“多渠道输入 + 会话化 Agent 调度 + 模型工具循环 + 记忆压缩系统”的轻量工作台。关键链路如下：

```text
Channel Adapter -> MessageBus -> AgentLoop -> ContextBuilder -> AgentRunner -> ToolRegistry / MCP -> Session / Memory / Consolidation
```

| 层级 | 主要角色 | 原生能力 | 对项目的价值 |
|---|---|---|---|
| 接入层 | Channel Adapter、MessageBus | CLI、WebUI、OpenAI-compatible API、聊天平台等入口统一成 `InboundMessage` / `OutboundMessage`。 | 多入口接入方便，但保险生产应收敛为企业受控客服入口。 |
| 调度层 | AgentLoop | 会话查找、恢复 checkpoint、命令判断、上下文构建、运行 Runner、保存历史。 | 适合作为单会话任务调度中心。 |
| 上下文层 | ContextBuilder | Identity、Bootstrap files、Tool contract、Memory、Active Skills、Skills、Recent History、Archived Context Summary。 | 对长任务、用户偏好、项目规则和 Skill 索引有清晰组装顺序。 |
| 执行层 | AgentRunner | 模型 + 工具循环、工具结果回填、microcompact、tool result budget、snip history。 | 对普通 Skill 全文、工具输出和长上下文有更细的治理机制。 |
| 工具层 | ToolRegistry、MCP | 内置工具、MCP stdio/HTTP、工具过滤、超时、schema 转换。 | 适合把 Dify Workflow、核心系统、规则服务封装成强类型 MCP 工具。 |
| 企业业务层 | 需新增 | 会话任务图、企业运行图、决策语境图谱、策略层、身份委托、审计事件、人工接管。 | Nanobot 原生不提供保险级治理，必须由项目架构补齐。 |

Nanobot 的强项是 Python 生态、MCP 接入、上下文治理、Skill `always`/普通索引分层和长任务压缩；局限是个人助理工具、文件/Shell/网页工具和长期记忆不能直接等同于保险生产能力。

### 1.4 Skill 的统一心智模型

两款底座都容易被误解为“Skill 插件被执行”。更准确的描述是：

> Skill 是给模型看的操作说明书，不是给运行时直接执行的插件。

两者的共同链路如下：

```mermaid
flowchart LR
    A["磁盘上的 Skill 文档"] --> B["启动或构建上下文时发现"]
    B --> C["解析 name / description / metadata"]
    C --> D["将 Skill 索引注入 system prompt"]
    D --> E["LLM 根据用户问题判断是否需要"]
    E --> F["LLM 调用 read/read_file 读取全文"]
    F --> G["Skill 全文作为 tool result 进入上下文"]
    G --> H["LLM 按说明调用真正工具或 MCP"]
    H --> I["工具结果回填给 LLM"]
    I --> J["历史变长后被压缩、裁剪或归档"]
    J --> K["需要时根据索引重新读取 Skill"]
```

因此，“Skill 运行机制”可以分为三层：

| 层级 | 内容 | 关键说明 |
|---|---|---|
| Prompt 层 | Skill 的名字、描述、路径、全文说明 | 告诉模型什么时候该用、怎么用、注意什么规则。 |
| Tool 层 | `read` / `read_file` / `exec` / `bash` / `edit` / MCP 工具 | 真正被运行时执行的能力，具备 schema、参数和执行函数。 |
| 企业治理层 | 策略、权限、审计、确认、状态机、SOP | 决定某个工具是否允许调用、调用结果是否可信、是否需要人工确认。 |

对保险客服场景，Skill 可以承载“处理某类业务的步骤说明”，但不能替代确定性的策略层和权限层。涉及客户状态、保单信息、理赔进度、权益变更时，模型必须通过受控工具重新查询权威系统，不能依赖长期记忆或旧上下文。

### 1.5 pi-mono 与 Nanobot Skill 生命周期差异

| 生命周期阶段 | pi-mono 机制 | Nanobot 机制 | 项目影响 |
|---|---|---|---|
| 发现路径 | 支持项目级 `.pi/skills/`、祖先目录 `.agents/skills/`、用户级 `~/.pi/skills` / `~/.agents/skills`、CLI skill path、additionalSkillPaths 等。既支持目录 `SKILL.md`，也支持根目录直接 `.md` 文件。 | 只扫描 `<workspace>/skills/<name>/SKILL.md` 和内置 `nanobot/skills/<name>/SKILL.md` 的直接子目录。 | pi-mono 更灵活，Nanobot 更规整。生产建议采用 Nanobot 类固定目录规范，降低治理成本。 |
| 优先级 | 先进入加载链路的 Skill 优先；同名冲突后加载者失效并记录诊断。单目录内部没有显式排序，不建议依赖文件系统顺序。 | workspace Skill 优先于 builtin Skill，同名 workspace 会 shadow builtin；被 `disabled_skills` 禁用后不会回退到 builtin。 | 需建立命名规范和发布流程，避免同名覆盖导致 always 行为或业务规则被意外替换。 |
| 元数据 | 重点解析 `name`、`description`、`disable-model-invocation`。`description` 缺失会导致 Skill 不加载或不可用于模型自主匹配。 | 重点解析 `description`、`always`、`metadata.nanobot.always`、`metadata.nanobot.requires.bins/env`，并兼容 `metadata.openclaw.*`。 | Nanobot 对常驻注入和依赖可用性表达更完整；pi-mono 的显式 `/skill:name` 更适合人工强触发。 |
| Prompt 注入 | 默认把全部启用 Skill 作为 `<available_skills>` XML 索引注入 system prompt，包含 name、description、location；普通情况下不注入全文。 | `always` Skill 全文注入 `# Active Skills`；普通 Skill 进入 `# Skills` 索引，包含 name、description、path、availability。 | 高优先级、短小、全局规则可考虑 always；多数业务 Skill 应保持普通索引，按需读取。 |
| 显式触发 | 支持 `/skill:name`，运行时先读取 Skill 正文，去掉 frontmatter，以 `<skill ...>` 包裹后直接注入当前 user message。 | 当前材料未显示等价的 `/skill:name` 强制展开入口；普通 Skill 主要依赖模型根据索引主动 `read_file`。`skill_names` 参数不等于自动加载。 | 对高风险流程，不应只依赖模型自主选择；应在企业路由层或 OpenClaw 状态机中明确触发。 |
| 自主触发 | 模型看到 `<available_skills>` 后，如果任务匹配 description，会调用 `read` 读取 location。 | 模型看到 `# Skills` 后，如果任务匹配 description，会调用 `read_file` 读取 `SKILL.md`。 | `description` 是模型选择 Skill 的关键入口，必须写清业务触发条件。 |
| 真正执行 | Skill 正文只是指令；模型继续调用 `read`、`bash`、`edit`、`write` 等工具。 | Skill 正文只是指令；模型继续调用 `read_file`、`exec`、`web_fetch`、MCP 等工具。 | 生产环境应禁用或替换本地 Shell/文件类工具，只暴露受控 MCP 业务工具。 |
| 单次读取限制 | `read` 工具有 2000 行 / 50KB 限制，超出需要 offset 继续读取。 | 超大 tool result 可落盘，仅在上下文保留路径与预览。 | Skill 正文不宜过长；大资料应拆到 `references/`，由 Skill 指导模型按需读取。 |
| 上下文压缩 | 会话 token 超过阈值时触发 session compaction，旧消息被总结成 `compactionSummary`，原始 Skill tool result 不再直接进入上下文。空闲压缩可能只保留最近合法消息。 | Runner 每次请求前对 `messages_for_model` 做 microcompact、tool result budget、snip；Session replay 和 consolidation 也会让旧 Skill 读取结果不再回放。 | 两者都不能保证普通 Skill 全文长期可见。模型需要时应重新读取 Skill。 |
| 恢复方式 | 只要 Skill 仍在索引里，模型可以再次根据 location 调用 `read`。 | 只要 Skill 仍在 `# Skills` 索引里，模型可以再次 `read_file(SKILL.md)`。 | 生产 Skill 应写成“可重读、可恢复”的短入口，不依赖跨轮记忆。 |
| Subagent | 当前材料强调 pi-coding-agent 不内置多 sub-agent 编排框架。 | subagent prompt 更轻，通常只拿 Skill summary，不注入主 agent 的 `# Active Skills` 全文。 | 多子任务并发需要企业层设计，不能默认认为 Skill 全文会自动传给子任务。 |

### 1.6 普通 Skill 的端到端运行时序

以“查询天气”或“新增 LLM provider”这类普通 Skill 为例，两款底座的运行语义高度相似：

1. **磁盘阶段**：Skill 文件存在于约定目录，正文是 Markdown 操作说明。

2. **发现阶段**：底座扫描目录，解析 `name`、`description`、路径和依赖。

3. **索引阶段**：system prompt 只放 Skill 名称、描述和路径；LLM 此时还看不到完整步骤。

4. **匹配阶段**：用户问题进入后，LLM 根据 Skill 描述判断是否相关。

5. **读取阶段**：LLM 调用 `read` 或 `read_file` 读取 Skill 全文。

6. **使用阶段**：Skill 全文作为 tool result 回填，LLM 按说明继续调用真正工具或 MCP 服务。

7. **短期复用阶段**：如果历史还短，下一轮可能仍能看到 Skill 全文，无需重读。

8. **压缩阶段**：旧 tool result 被 summary、占位符或外置文件替代。

9. **裁剪阶段**：旧 Skill 读取记录从当前模型上下文中完全消失。

10. **重读阶段**：后续再次需要时，LLM 根据 system prompt 中仍然存在的索引重新读取 Skill。

这说明普通 Skill 的生命周期不是“加载一次，永久记住”，而是：

```text
发现 -> 索引 -> 读取 -> 使用 -> 保存 -> 压缩 -> 消失 -> 重新读取
```

### 1.7 Skill 消失、压缩与重读机制

这里的“消失”不是删除磁盘上的 `SKILL.md`，而是指下一次发给 LLM 的上下文里看不到 Skill 全文。

| 消失类型 | pi-mono | Nanobot | 应对方式 |
|---|---|---|---|
| 发现/索引阶段不可见 | 路径未被加载、frontmatter 不合规或 `description` 缺失会导致 Skill 不可用；`disable-model-invocation` 只会排除模型自主调用，不影响显式 `/skill:name`。 | 路径不是直接子目录、文件名不是 `SKILL.md`、被 `disabled_skills` 禁用、workspace 同名 shadow builtin。 | 建立目录、命名、frontmatter、禁用清单检查。 |
| 从全文常驻降级为索引 | 默认无 Nanobot 式 always 全文注入，主要都是索引。 | `always` 缺失或 requirements 不满足时，不进 `# Active Skills`，但可进入 `# Skills` 并标注 unavailable。 | 只把短小、全局、稳定的规则设为 always。 |
| 从索引消失但仍可见 | 不适用或较少见。 | always Skill 会从 `# Skills` 索引排除，因为已经在 `# Active Skills` 全文出现。 | 这是正常去重，不是禁用。 |
| 读取后被压缩 | `read` 的 Skill tool result 进入旧历史后，可能被 compaction summary 替代。 | 旧 `read_file(SKILL.md)` tool result 可能被 microcompact 成 `[read_file result omitted from context]`。 | 在全局 Skill 模板或描述中提示：前文不可见时应重读。 |
| 读取后被裁剪 | 多轮后旧消息被 session compaction、idle auto-compact 或最近消息窗口挤出。 | `_snip_history`、`Session.get_history`、`last_consolidated`、idle auto-compact 都可能让旧 Skill 记录不再回放。 | 把关键流程放在 Skill 开头，允许快速重读恢复。 |
| 语义被摘要稀释 | 多次 compaction 后 summary 可能只保留“曾使用某 Skill”，不保留细节。 | consolidation 可能把旧操作写成归档摘要，但不保留原始 tool call 和全文。 | 高风险任务每次执行前重新读取当前版本 Skill 和权威工具结果。 |

对业务 Skill 编写的直接要求是：

1. `description` 必须清楚说明触发场景、业务对象和风险边界。
2. SKILL.md 开始部分应能快速恢复核心流程、适用条件、禁止事项和输出格式。
3. 大量背景资料放入 `references/`，确定性操作放入 `scripts/` 或企业 MCP 工具。
4. 不假设模型跨轮仍记得 Skill 全文。
5. 对高风险 Skill，应由状态机或业务路由明确触发，而不是完全依赖模型自主判断。

### 1.8 自主智能体范式对比：千帆自由规划 / React / OpenClaw 类 Skill 驱动

项目最终要解决的不是“有没有 Agent Loop”，而是“复杂业务中谁控制主流程”。三类范式差异如下：

| 范式 | 主流程控制者 | 工具/Skill 角色 | 优势 | 主要风险 | 适配场景 |
|---|---|---|---|---|---|
| 百度千帆自主智能体 | 云平台与大模型自主规划 | 工具作为平台组件被模型自由选择 | 开箱快、配置成本低、适合演示和轻量问答 | 黑盒程度高、路径随机、私有化和审计边界弱 | 通用问答、营销演示、无严格 SOP 的轻量场景 |
| React 范式 | LLM 在 `Thought -> Action -> Observation` 循环中逐步决定 | 工具被动执行模型每一步 action | 通用性强，可处理开放任务 | 链路长、步骤漂移、重复调用、复杂业务不可控 | 非标准探索、开发辅助、开放式研究任务 |
| OpenClaw 类 Skill 驱动 | 企业状态机、规则引擎和 Skill 编排控制主干；LLM 处理局部语义 | Skill 承载流程说明和局部规则，工具/MCP 执行确定动作 | 主干可控、可审计、可复用，兼顾模型理解能力 | 新场景需要维护编排和规则；极致自由度低于 React | 保险、合规、制度、理赔、投诉等强约束业务 |

OpenClaw 类机制的核心不是把 LLM 关死，而是分层：

1. **底座负责运行**：模型调用、工具调用、上下文、会话、流式输出和压缩。

2. **状态机负责主干**：识别意图、缺失信息、可调用工具、下一步动作、失败处理和转人工条件。

3. **Skill 负责说明**：告诉模型某类任务的步骤、证据要求、参数填充规则和输出格式。

4. **策略层负责授权**：决定工具是否可执行、是否需要客户确认、是否需要人工审核。

5. **审计层负责追踪**：记录输入、证据、工具参数摘要、策略版本、结果和人工介入。

### 1.9 企业落地建议

| 建议项 | 具体建议 |
|---|---|
| 底座选择 | 若项目优先 Python、MCP、图谱和数据工程集成，可优先采用 Nanobot 作为轻量 Agent Loop；若项目优先 TypeScript、事件流、SDK/RPC 和开发者产品化，可参考 pi-mono。 |
| Skill 策略 | 普通业务 Skill 采用“索引常驻 + 正文按需读取”；只有短小、全局、每轮都必须遵守的规则才使用 Nanobot `always`。 |
| 工具策略 | 禁止把本地 Shell、文件系统、网页抓取等个人助理工具直接暴露给保险生产模型；统一封装为强类型 MCP 工具或受控业务 API。 |
| 流程策略 | 使用 OpenClaw 类状态机控制主干流程，LLM 只在单步骤内做语义理解、参数补全、证据组织和自然语言表达。 |
| 上下文策略 | 普通 Skill 全文随时可能被压缩或裁剪，所有高风险任务执行前必须重新读取当前 Skill，并重新查询权威系统。 |
| 审计策略 | 把 Agent turn、工具提议、策略决策、MCP 调用、证据引用和人工确认写成追加式审计事件。 |
| 知识策略 | Skill 不承载业务事实真相；保险条款、版本、客户状态和理赔进度必须来自 RAG、GraphRAG、业务系统或规则服务。 |

**本节最终结论**：pi-mono 与 Nanobot 都能提供 Agent Loop、工具调用和 Skill 渐进式加载能力，但它们都不是开箱即用的保险级智能体平台。项目应采用“底座 + OpenClaw 类 Skill 编排 + 语境图谱 RAG + 策略审计层”的组合路线：底座提供运行能力，Skill 提供可重读流程说明，企业层提供确定性治理。

## 二、RAG 检索体系对比：标准 RAG / 知识图谱 RAG / 语境图谱 RAG

### 2\.1 本节结论与选型口径

本节将 RAG 检索体系分为三层能力：**标准 RAG 解决“找相似文本”**，**知识图谱 RAG 解决“找实体关系和多跳路径”**，**语境图谱 RAG 解决“在当前任务、时间、版本、权限和证据约束下找可用知识”**。

| 结论项 | 判断 | 对项目选型的含义 |
|---|---|---|
| 标准 RAG | 成熟、成本低、落地快，但核心是分块文本相似度召回。 | 适合 FAQ、简单制度查询、低风险知识库；不适合复杂规则、多版本、多条件问答作为唯一方案。 |
| 知识图谱 RAG | 在文本检索之外引入实体、关系、路径和图数据库。 | 适合解决跨文档串联、实体消歧、关系推理、多跳查询，是从“文档片段”升级到“结构化知识”的关键层。 |
| 语境图谱 RAG | 在知识图谱之上增加时间、来源、证据、任务、权限、状态和决策链。 | 适合保险、法务、合规、客服等强规则场景，是解决“同一问题在不同语境下答案不同”的核心方案。 |
| 技术归属 | 三者都可以基于开源技术栈实现；语境图谱不是单一闭源产品，而是图数据库、RAG、事件日志、权限过滤和业务规则的组合架构。 | 项目可以自研组合，不存在被某个厂商或某个 IDEA 独占的技术壁垒。 |
| 推荐路线 | 一期标准 RAG 可作为基础召回层保留；二期应建设“向量检索 \+ 知识图谱 \+ 语境过滤”的进阶 RAG。 | 对核心业务问答，最终应以语境图谱 RAG 作为目标形态，知识图谱 RAG 作为中间演进层。 |

### 2\.2 三者核心定义与本质区别

| 对比维度 | 标准 RAG（分块文本检索） | 知识图谱 RAG（Knowledge Graph RAG / GraphRAG） | 语境图谱 RAG（Context Graph RAG） |
|---|---|---|---|
| 核心定位 | 从文档块中找相似内容，再交给大模型总结。 | 将文档和业务数据抽取为实体、关系、路径，用图结构增强检索。 | 将事实、关系、时间、来源、证据、任务、权限和决策过程一起组织成上下文子图。 |
| 主要回答 | “哪段文本和问题最像？” | “哪些实体、关系和路径与问题相关？” | “当前用户、当前时间、当前任务下，哪些证据和规则可用？” |
| 数据形态 | 文本 Chunk、向量、关键词索引。 | 实体、关系、属性、文档 Chunk、图数据库。 | 实体、关系、事件、证据、时间、版本、权限、会话、任务、状态。 |
| 典型存储 | 向量库、全文检索、对象存储。 | 向量库 \+ 图数据库，例如 Neo4j、NebulaGraph、JanusGraph、GraphDB。 | 图数据库 \+ 向量库 \+ 事件日志 \+ 权限/规则服务；可复用知识图谱底座。 |
| 推理能力 | 弱，主要依赖 LLM 在召回片段上做总结。 | 中等到强，支持实体消歧、多跳路径、关系推理。 | 更强，支持多跳关系 \+ 时间过滤 \+ 来源校验 \+ 权限过滤 \+ 场景匹配。 |
| 可解释性 | 可引用文档片段，但关系路径弱。 | 可解释实体关系路径。 | 可解释“为什么在当前语境下得出该结论”，包括证据、版本、规则和决策链。 |
| 适用问题 | 简单问答、说明类查询、宽泛知识检索。 | 跨文档关联、实体关系查询、复杂路径、知识融合。 | 多版本规则、时效约束、身份差异、场景化流程、审计追踪、Agent 记忆。 |
| 主要风险 | 召回片段碎片化、语义相似但业务不相关、幻觉偏高。 | 图谱建模和实体对齐成本高，静态事实容易缺少时间/来源/场景。 | 建模更复杂，需要治理时间、来源、权限、生命周期和上下文写回。 |

一句话概括：

```text
标准 RAG = 文本相似度召回
知识图谱 RAG = 文本召回 + 实体关系路径
语境图谱 RAG = 文本召回 + 实体关系路径 + 时间/来源/权限/任务/证据/决策链
```

### 2\.3 知识图谱 RAG 与语境图谱 RAG 的关系

知识图谱 RAG 不是被语境图谱 RAG 否定，而是后者的基础层。二者关系可以理解为：

```text
文档 / 数据库 / 业务系统
        ↓
实体抽取、关系抽取、实体对齐
        ↓
知识图谱：稳定实体、概念、事实、关系
        ↓
语境图谱：时间、来源、证据、任务、权限、状态、决策链
        ↓
RAG / Agent / 审计 / 业务流程自动化
```

知识图谱 RAG 主要解决：

1. 文档碎片之间没有结构化关联。
2. 同一实体在不同文档中名称不一致。
3. 问题需要跨文档、多跳关系才能回答。
4. 需要输出实体之间的关系路径。

语境图谱 RAG 在此基础上进一步解决：

1. 事实缺少时间和版本，导致新旧规则混用。
2. 事实缺少来源和证据，导致回答无法审计。
3. 同一实体或条款在不同业务场景下结论不同。
4. 当前用户权限、客户身份、任务状态会影响可用答案。
5. Agent 调用工具、形成结论、产生动作后缺少决策链追踪。

因此，二期建设不应把“知识图谱”和“语境图谱”看成二选一，而应采用分层方案：**知识图谱做稳定事实层，语境图谱做上下文和决策层**。

### 2\.4 技术栈 \& 开源属性论证

#### 2\.4\.1 标准 RAG

- **核心组件**：文档解析、文档切片、Embedding 向量模型、向量数据库、BM25 / 关键词检索、Rerank 重排模型、LLM 生成；
- **主流框架**：LangChain、LlamaIndex、Dify、Haystack、Qdrant、Milvus、Weaviate、Elasticsearch / OpenSearch、BGE / E5 / Jina Embeddings 等；
- **技术属性**：生态成熟、开源组件丰富、部署难度低，是项目一期已经采用的基础方案；
- **能力边界**：仅靠分块文本无法稳定表达实体关系、规则约束、版本差异、用户身份和任务状态。

#### 2\.4\.2 知识图谱 RAG

知识图谱 RAG 是通用开源技术路线，不属于某个厂商或某个 IDEA 独有能力。其核心由三类组件组成：

| 组件 | 可选技术 | 作用 |
|---|---|---|
| 图数据库 | Neo4j、NebulaGraph、JanusGraph、TigerGraph、Memgraph、GraphDB、Stardog、Amazon Neptune | 存储实体、关系、属性和路径，支持图遍历和多跳查询。 |
| 图谱构建 | spaCy、OpenNRE、DeepKE、LLM Prompt 抽取、规则抽取、ETL 映射 | 从文档和业务数据中抽取实体、关系、属性和三元组。 |
| RAG 编排 | Microsoft GraphRAG、LlamaIndex Knowledge / Property Graph Index、LangChain GraphQA、Neo4j GraphRAG | 将向量召回、图查询、重排和 LLM 回答串联起来。 |

关键结论：

1. 知识图谱本身是成熟领域，RDF、OWL、SPARQL、Neo4j、Cypher、Gremlin 等都有长期生态。
2. GraphRAG 是知识图谱与 RAG 的组合，不是闭源专利式方案。
3. 项目可以基于开源图数据库和开源 RAG 框架独立搭建、二次开发、私有化部署。

#### 2\.4\.3 语境图谱 RAG

语境图谱 RAG 不是一个统一标准产品，而是一种工程架构：在知识图谱 RAG 之上，增加上下文建模和条件过滤能力。

| 能力层 | 可选技术 | 作用 |
|---|---|---|
| 语境建模 | RDF Named Graph、PROV-O、OWL-Time、RDF-star、属性图边属性、事件图模型 | 表达时间、来源、证据、有效期、置信度、权限和上下文边界。 |
| 图检索 | Neo4j、Memgraph、FalkorDB、NebulaGraph、GraphDB、Stardog | 支持按实体、关系、时间、权限和路径做条件查询。 |
| GraphRAG 编排 | Microsoft GraphRAG、Neo4j GraphRAG、LlamaIndex PropertyGraphIndex、LangChain \+ Neo4j | 将图谱召回结果转换为 LLM 可用上下文。 |
| Agent 记忆与状态 | Graphiti / Zep、Cognee、Mem0、LangGraph Persistence、OpenTelemetry Trace | 记录会话、任务、工具调用、用户偏好、决策链和运行时上下文。 |
| 业务规则层 | 自研规则服务、权限服务、策略引擎、审批流、MCP 工具网关 | 负责保险业务中的身份、渠道、产品版本、地区、时间和合规约束。 |

关键结论：

1. 语境图谱的底层能力可完全复用开源图数据库、RAG 框架和观测组件。
2. 真正需要项目自研的是业务语境模型，例如保险产品版本、客户身份、地区规则、理赔状态、权限边界和审计口径。
3. 语境图谱 RAG 的壁垒不在底层组件，而在业务知识建模、数据治理和规则语境沉淀。

### 2\.5 检索流程对比

#### 2\.5\.1 标准 RAG 流程

```text
文档解析
  ↓
固定分块
  ↓
向量化入库 + 关键词索引
  ↓
向量/BM25 多路召回
  ↓
Rerank 重排
  ↓
拼接文本片段
  ↓
LLM 总结回答
```

适合简单说明类问题，但对“条款之间的关系”“新旧版本冲突”“不同用户身份适用规则不同”等问题支持不足。

#### 2\.5\.2 知识图谱 RAG 流程

```text
文档/结构化数据
  ↓
实体抽取 + 关系抽取 + 实体对齐
  ↓
向量库 + 图数据库双存储
  ↓
问题理解 + 实体链接
  ↓
文本召回 + 图谱多跳查询
  ↓
路径/证据融合重排
  ↓
LLM 基于文本和关系路径回答
```

适合解决“跨文档、多实体、多跳关系”的问题。例如客户、保单、产品、条款、理赔材料之间存在关联时，图谱能帮助系统找到完整关系链。

#### 2\.5\.3 语境图谱 RAG 流程

```text
文档/业务数据/会话/工具调用/日志
  ↓
抽取实体、关系、事件、时间、来源、权限、置信度
  ↓
构建知识图谱 + 语境图谱
  ↓
解析用户问题中的身份、时间、渠道、产品版本、任务状态
  ↓
向量召回 + 图遍历 + 时间过滤 + 权限过滤 + 规则过滤
  ↓
构造当前任务可用的上下文子图
  ↓
LLM 基于证据链和语境约束回答
  ↓
回答、引用、工具调用结果和用户反馈写回语境图谱
```

适合保险制度、理赔咨询、合规问答等强约束场景。其目标不是“召回更多内容”，而是筛掉当前语境下不可用、过期、无权限或缺证据的内容。

### 2\.6 业务例子：同一个问题在三类 RAG 下的差异

假设用户问题是：

```text
客户购买了产品 X，现在想申请某项理赔，是否符合条件？
```

| 检索体系     | 可能处理方式                                                 | 主要风险或优势                                             |
| -------- | ------------------------------------------------------ | --------------------------------------------------- |
| 标准 RAG   | 根据“产品 X”“理赔条件”等关键词和向量相似度召回相关条款片段，由 LLM 总结。             | 可能召回旧版本条款、相似但不适用条款；难以判断客户身份、保单状态和事故时间是否满足条件。        |
| 知识图谱 RAG | 找到客户、保单、产品 X、责任条款、免责条款、理赔材料之间的关系路径。                    | 能回答“相关条款有哪些、实体之间如何关联”，但如果没有语境层，仍可能忽略版本、时间、地区、渠道和权限。 |
| 语境图谱 RAG | 在关系路径基础上继续过滤：保单生效时间、等待期、事故时间、条款版本、客户身份、地区规则、材料状态、当前权限。 | 能给出更接近业务判断的回答，并说明依据、版本、证据和仍需人工确认的点。                 |

进一步对比：

```text
标准 RAG：这段条款看起来相关。
知识图谱 RAG：客户 A 的保单 B 与产品 X、条款 C、材料 D 有关系。
语境图谱 RAG：在客户 A 当前保单状态、事故时间、条款版本和理赔材料条件下，条款 C 当前适用，但仍需补充材料 D。
```

### 2\.7 优势、劣势与边界

| 方案 | 优势 | 劣势 | 适合阶段 |
|---|---|---|---|
| 标准 RAG | 成熟、便宜、上线快、维护简单。 | 结构化能力弱，难以处理多跳、版本、权限和规则约束。 | 一期基础知识库、非核心低风险问答。 |
| 知识图谱 RAG | 实体关系清晰，支持多跳查询和路径解释，能把碎片文档变成结构化知识。 | 建模、抽取、实体对齐和图数据库运维成本更高；若只记录静态事实，仍容易忽略语境。 | 二期知识增强、复杂实体关系和跨文档问答。 |
| 语境图谱 RAG | 能处理时间、版本、来源、权限、任务状态和决策链，适合强规则场景和 Agent。 | 建模最复杂，需要持续治理上下文生命周期、权限、证据和写回机制。 | 核心业务问答、保险/法务/合规、Agent 记忆和审计。 |

### 2\.8 本节最终结论

1. **标准 RAG 是基础能力，不应完全废弃**：它适合做粗召回和简单问答，也是后续图谱 RAG 的文本证据来源。
2. **知识图谱 RAG 是必要升级层**：它解决传统 RAG 的碎片化问题，把文档和业务数据组织成实体、关系和路径。
3. **语境图谱 RAG 是核心目标层**：它解决知识图谱缺少时间、来源、证据、权限、任务场景和决策过程的问题。
4. **项目推荐采用组合架构**：`标准 RAG 粗召回 + 知识图谱关系推理 + 语境图谱条件过滤 + LLM 生成 + 审计写回`。
5. **对保险类强规则场景，语境图谱 RAG 的价值最高**：它能显著降低新旧规则混淆、条款误用、身份误判和无依据回答的风险。

## 三、新架构 vs 项目一期传统方案：能力差异、提升与优势

### 3\.1 本节结论与升级判断

项目一期采用 **Dify Workflow \+ 传统 RAG** 实现智能客服，本质是“固定工作流 \+ 文档片段检索 \+ 大模型总结”。该方案适合快速搭建基础知识库问答，但在保险客服、理赔咨询、制度解释、合规问答等强规则场景中，会逐渐暴露出结构化知识不足、规则语境不足、复杂任务承载不足和审计解释不足的问题。

新架构 `pi / Nanobot + 知识图谱 / 语境图谱` 的优势不在于单纯替换 Dify，而是将系统从：

```text
一期：从文档里找相似片段，让大模型总结
```

升级为：

```text
新架构：先理解任务，再按规则、图谱、上下文和工具链完成受控回答
```

因此，新架构不是对一期能力的否定，而是对一期能力的增强。推荐策略是保留一期标准 RAG 作为文本证据召回层，同时引入知识图谱解决实体关系和多跳问题，引入语境图谱解决时间、版本、权限、证据和决策链问题，再由 pi / Nanobot 承载 Agent Loop、Skill、工具调用和审计治理。

### 3\.2 项目一期方案现状与核心短板

项目一期方案可以概括为：

```text
用户问题 -> Dify Workflow -> 文档分块检索 -> 向量/BM25 召回 -> 文本片段拼接 -> LLM 总结回答
```

其优势是上线快、配置简单、适合基础 FAQ 和说明类知识问答。但在智能客服进入真实业务场景后，主要短板如下：

| 一期短板 | 具体表现 | 对业务的影响 |
|---|---|---|
| 文档片段碎片化 | 检索对象主要是 Chunk，缺少实体、条款、产品、保单之间的结构化关系。 | 多文档、多条款、多实体问题容易答不完整。 |
| 语义相似不等于业务适用 | 相似度高的片段可能来自旧版本、其他产品、其他地区或其他客户身份。 | 容易出现“看起来相关，但实际不适用”的回答。 |
| 多跳推理能力弱 | 难以稳定串联客户、保单、产品、责任条款、免责条款、理赔材料和流程状态。 | 理赔类、保单类、规则类复杂问题准确率不足。 |
| 规则和版本治理弱 | 文档中存在生效时间、失效时间、产品版本、地区规则、渠道差异时，普通 RAG 很难自动区分。 | 容易新旧规则混用、条款版本混用。 |
| 多轮任务状态弱 | Workflow 更适合预设流程，不擅长根据工具结果和用户补充信息动态推进任务。 | 用户体验停留在“问答机器人”，难升级为“业务办理助手”。 |
| 可解释和审计不足 | 通常只能说明引用了哪些文档片段，难以说明为什么在当前场景下这样判断。 | 投诉复盘、合规审计、人工复核成本较高。 |
| 扩展维护压力 | 业务复杂后节点、分支、Prompt 和规则容易散落在 Workflow 内。 | 后续维护难、复用弱、问题定位慢。 |

### 3\.3 新整体架构定位

新架构建议采用分层方式，而不是把所有能力塞进一个 Workflow：

```text
pi / Nanobot：Agent Loop、工具调用、上下文管理
Skill：业务 SOP、操作说明、话术规范、人工确认规则
知识图谱：客户、保单、产品、条款、材料、规则之间的稳定关系
语境图谱：时间、版本、权限、证据、任务状态、决策链
MCP / 工具层：保单查询、理赔进度、客户信息、规则服务、工单系统
审计层：记录输入、检索、工具调用、规则命中、回答依据
```

其中：

1. **pi / Nanobot** 负责智能体运行时能力，包括多轮对话、工具调用、上下文治理、事件流和 Skill 编排。
2. **知识图谱** 负责将碎片化文档升级为结构化知识，表达实体、关系、条款、产品、材料和业务对象之间的路径。
3. **语境图谱** 负责判断知识在当前任务中是否可用，表达时间、版本、来源、权限、证据、客户身份、任务状态和决策过程。
4. **MCP / 工具层** 负责连接真实业务系统，让智能客服不止能回答，还能查询状态、补充材料、创建工单和触发人工转接。

### 3\.4 综合能力对比

| 能力维度 | 一期 Dify Workflow \+ 传统 RAG | 新架构 pi / Nanobot \+ 知识图谱 / 语境图谱 |
|---|---|---|
| 核心模式 | 固定工作流 \+ 文档片段检索。 | Agent Loop \+ Skill 编排 \+ 图谱检索 \+ 受控工具调用。 |
| 知识形态 | 文本 Chunk 为主，结构化关系弱。 | 文本 Chunk \+ 实体 \+ 关系 \+ 条款 \+ 证据 \+ 时间 \+ 状态。 |
| 检索方式 | 向量/BM25 相似度召回。 | 向量召回 \+ 图谱多跳 \+ 时间过滤 \+ 权限过滤 \+ 规则过滤。 |
| 复杂推理 | 主要依赖 LLM 对召回片段做总结。 | 先通过图谱确定实体关系和路径，再由 LLM 基于证据生成。 |
| 规则适配 | 难以稳定区分时间、版本、地区、渠道、客户身份。 | 可按生效期、失效期、产品版本、地区、渠道、身份和任务状态过滤。 |
| 多轮任务 | 流程固定，状态表达有限。 | 支持会话状态、任务状态、工具结果和必要的长期记忆。 |
| 工具接入 | 工作流节点接工具，复杂后分支膨胀。 | 通过 MCP / Tool Registry 接核心系统，并可在调用前后做策略拦截。 |
| 可解释性 | 主要引用文档片段。 | 可解释实体路径、证据来源、条款版本、规则命中和决策链。 |
| 可审计性 | 依赖 Workflow 日志和回答引用。 | 可记录输入、检索、图路径、工具调用、权限过滤、规则命中和输出依据。 |
| 扩展性 | 新业务往往新增流程、节点和 Prompt。 | 底座、Skill、图谱、工具、审计分层扩展，可跨业务线复用。 |
| 适用边界 | FAQ、简单知识问答、低风险说明类场景。 | 保险客服、理赔咨询、合规问答、复杂流程办理、人工协同场景。 |

### 3\.5 关键能力提升

#### 1\. 检索能力：从“找相似文本”到“找可用知识”

传统 RAG 最大的问题是：**语义相似不等于业务适用**。

例如用户问：

```text
这个产品能不能赔？
```

一期可能召回一段看起来相似的理赔条款，但不一定校验产品版本、保单状态、事故时间、等待期、客户身份、地区规则、免责条款和当前材料是否齐全。

新架构可以将问题拆成多个业务判断：

```text
客户是谁？
保单是否有效？
产品版本是什么？
事故时间是否在保障期内？
当前条款是否已过期？
是否命中免责规则？
需要哪些材料？
回答依据是哪条证据？
```

这意味着新架构不是召回更多文本，而是召回当前语境下真正可用的知识。

#### 2\. 推理能力：从“文本总结”到“关系路径判断”

一期方案更像单轮问答流水线，不擅长多跳关系：

```text
客户 -> 保单 -> 产品 -> 责任条款 -> 免责条款 -> 理赔材料 -> 当前流程状态
```

知识图谱可以解决“关系链”问题，语境图谱进一步解决“这条关系链当前是否有效”的问题。

示例：

```text
知识图谱：客户 A 购买了产品 X，产品 X 包含责任条款 C。
语境图谱：客户 A 的保单当前有效，事故发生在等待期后，条款 C 属于 2025 版，当前仍生效，但材料 M 缺失。
```

因此，新架构可以从“回答知识”升级为“辅助业务判断”。

#### 3\. 规则能力：从“召回条款”到“按语境过滤条款”

保险客服中的高频错误往往不是模型不会总结，而是模型混用了规则：

```text
新旧规则混用
不同产品版本混用
不同地区规则混用
不同客户身份混用
历史条款和现行条款混用
```

语境图谱可以给知识增加生效时间、失效时间、产品版本、适用地区、适用渠道、客户身份、来源文档、审核状态和权限等级。回答前先过滤掉“不适用于当前场景”的知识，可以显著降低条款误用和规则误判。

#### 4\. 执行能力：从“问答流程”到“任务办理”

Dify Workflow 适合配置确定性流程，但复杂任务会出现流程固定、分支膨胀、上下文传递弱、工具调用治理有限等问题。

pi / Nanobot 这类底座更适合作为 Agent Runtime：

| 能力 | 新架构价值 |
|---|---|
| Agent Loop | 支持多轮规划、工具调用、观察结果、继续执行。 |
| Skill 机制 | 将业务 SOP、客服话术、理赔流程、人工确认规则做成可复用说明。 |
| Tool / MCP 接入 | 可接保单查询、理赔进度、客户信息、规则服务、工单系统。 |
| 上下文治理 | 控制哪些历史、记忆、工具结果和证据进入当前推理。 |
| 事件流 / Hook | 便于做审计、拦截、人工确认、异常处理和合规留痕。 |

新架构不是让模型自由发挥，而是让模型在受控工具、受控流程、受控知识中完成任务。

#### 5\. 审计能力：从“引用片段”到“可解释决策链”

一期通常只能解释：

```text
我引用了哪些文档片段。
```

业务真正需要的是：

```text
为什么这样判断？
用了哪个版本的条款？
依据来自哪里？
是否经过权限过滤？
是否触发了免责规则？
哪些信息不确定，需要人工确认？
```

语境图谱可以记录：

```text
用户问题 -> 命中实体 -> 相关保单 -> 适用条款 -> 证据片段 -> 规则过滤 -> 最终回答
```

这对线上客服、投诉复盘、合规审计和人工复核非常关键。

### 3\.6 可解决的一期核心问题

| 一期问题 | 新架构解决方式 | 业务价值 |
|---|---|---|
| 文档片段碎片化 | 知识图谱建立客户、保单、产品、条款、材料之间的实体关系。 | 提升跨文档、跨条款问答的完整性。 |
| 多跳问题答不好 | 图谱路径查询和关系推理。 | 支持复杂理赔、保单、规则链路判断。 |
| 新旧条款混用 | 语境图谱增加版本、生效期、失效期。 | 降低规则误用风险。 |
| 相似但不适用 | 按产品、身份、地区、渠道、任务状态过滤。 | 提升回答与当前用户场景的匹配度。 |
| 回答不可审计 | 保存证据、来源、规则命中和决策链。 | 支持投诉复盘、合规检查和人工复核。 |
| 工作流复杂后难维护 | Agent \+ Skill \+ 工具 \+ 图谱分层编排。 | 降低流程分支膨胀和重复建设。 |
| 无法办理复杂任务 | Agent 可多步调用工具、查询状态、推进流程。 | 从知识问答升级为业务办理助手。 |
| 幻觉偏高 | 先结构化约束，再基于证据生成。 | 降低无依据回答和错误承诺。 |
| 人工复核成本高 | 输出依据、不确定点和人工介入条件。 | 提高坐席辅助效率，减少重复核查。 |
| 业务能力难复用 | Skill、图谱、工具可跨业务线复用。 | 支持后续产品线和场景扩展。 |

### 3\.7 典型业务效果示例

以理赔咨询为例，用户问：

```text
我这个情况能不能理赔？
```

一期方案可能回答一段泛化条款：

```text
根据产品条款，符合某些条件时可以申请理赔，具体以审核为准。
```

新架构可以结合保单、条款、事故时间、材料状态和当前流程，给出更接近业务办理的回答：

```text
根据你当前保单状态、事故时间和产品条款，初步看符合责任范围；
但系统显示缺少医院诊断证明和费用明细，补齐后可进入人工审核。
```

二者差异在于：

| 方案 | 回答重点 |
|---|---|
| 一期传统 RAG | 解释“条款大概怎么说”。 |
| 新架构 | 判断“当前客户、当前保单、当前材料状态下下一步该怎么做”。 |

这就是从“知识问答机器人”升级为“业务办理助手”的关键差异。

### 3\.8 推荐演进路线

新架构不建议推倒重来，而应分层演进：

```text
保留一期标准 RAG 作为文本召回基础
        ↓
引入知识图谱解决实体关系和多跳问题
        ↓
引入语境图谱解决时间、版本、权限、证据和决策链
        ↓
用 pi / Nanobot 承载 Agent Loop、Skill、工具调用和审计治理
```

| 阶段 | 目标 | 关键建设内容 |
|---|---|---|
| 阶段 1 | 保留一期能力，补齐评测和日志。 | 继续使用 Dify RAG，建立问题集、错误类型、召回日志和人工标注。 |
| 阶段 2 | 引入知识图谱。 | 抽取产品、条款、保单、材料、免责规则等实体关系，支持多跳查询。 |
| 阶段 3 | 引入语境图谱。 | 增加版本、生效期、失效期、来源、权限、客户身份、任务状态。 |
| 阶段 4 | 引入智能体底座。 | 用 pi / Nanobot 接管复杂任务编排、工具调用、Skill、审计和人工确认。 |

### 3\.9 本节最终结论

如果只是简单知识库问答，一期 Dify Workflow \+ 传统 RAG 足够。但如果要做真正可用于保险业务的智能客服，新架构更有价值：

```text
Dify Workflow + 传统 RAG 解决“能不能答”
pi / Nanobot + 知识图谱 / 语境图谱解决“能不能答对、能不能解释、能不能办理、能不能审计”
```

最终目标是把智能客服从“文档问答系统”升级为“可控、可解释、可审计、可办理业务的智能体系统”。

## 四、整体结论

### 4\.1 总体判断

项目一期采用 Dify Workflow \+ 传统 RAG，适合快速完成基础知识库问答建设。但如果下一阶段目标是面向保险客服、理赔咨询、制度解释、合规问答等强规则场景，继续只依赖固定 Workflow 和分块文本检索，会难以解决多跳关系、条款版本、时效约束、客户身份、材料状态、权限边界和审计解释等问题。

因此，推荐项目下一阶段采用：

```text
pi / Nanobot 智能体底座
        +
OpenClaw 类 Skill 驱动编排
        +
标准 RAG 粗召回
        +
知识图谱关系推理
        +
语境图谱条件过滤
        +
MCP 工具网关与审计治理
```

该路线不是推倒一期，而是在一期 RAG 基础上做结构化增强、语境增强和智能体编排增强。

### 4\.2 分层结论

| 层面 | 结论 | 对项目的意义 |
|---|---|---|
| 智能体底座 | 不建议用纯模型自由规划或通用 ReAct 直接承载保险核心流程，应采用 OpenClaw 类 Skill 驱动机制。 | 让模型在受控 SOP、受控工具、受控状态机中执行，降低不可控行为。 |
| pi / Nanobot 选择 | Nanobot 更适合作为 Python 轻量 Agent Loop、MCP 接入和上下文治理底座；pi-mono 可作为 TypeScript Runtime、事件流和工具治理参考。 | 二者都不是开箱即用的保险平台，必须叠加企业级策略层、权限层和审计层。 |
| Skill 机制 | Skill 本质是给 LLM 阅读的操作说明书，不是直接执行的插件。 | 适合沉淀客服 SOP、理赔流程、工具调用规范、人工确认条件和话术规则。 |
| RAG 检索 | 标准 RAG 适合作为文本粗召回层，但不应作为核心业务问答的唯一方案。 | 继续复用一期成果，同时补齐结构化关系和语境过滤。 |
| 知识图谱 | 知识图谱 RAG 解决实体、条款、产品、保单、材料之间的结构化关系和多跳查询。 | 从“文档片段检索”升级为“业务关系检索”。 |
| 语境图谱 | 语境图谱 RAG 在知识图谱之上增加时间、版本、来源、权限、客户身份、任务状态和决策链。 | 解决保险场景中“同一条款在不同语境下答案不同”的核心问题。 |
| 工具接入 | 通过 MCP / Tool Registry 接保单、理赔、客户、工单、规则服务。 | 让客服从“问答机器人”升级为“可查询、可推进、可转人工的业务助手”。 |
| 审计治理 | 需要记录输入、检索、图路径、工具调用、规则命中、回答依据和人工介入点。 | 支撑投诉复盘、合规检查、质量评估和持续优化。 |

### 4\.3 相比一期的核心提升

新架构相较一期 Dify Workflow \+ 标准 RAG，核心提升可以概括为五点：

1. **知识能力升级**：从文本 Chunk 检索升级为文本、实体、关系、条款、证据、时间和状态共同检索。
2. **推理能力升级**：从 LLM 对片段做总结，升级为图谱路径查询 \+ 语境过滤 \+ 基于证据生成。
3. **规则能力升级**：从召回相似条款，升级为按生效期、失效期、产品版本、地区、渠道、客户身份和材料状态过滤条款。
4. **执行能力升级**：从固定问答流程，升级为 Agent Loop \+ Skill \+ MCP 工具调用，支持复杂任务办理。
5. **治理能力升级**：从引用文档片段，升级为可追踪证据来源、规则命中、权限过滤、工具调用和决策链。

最终效果是：

```text
Dify Workflow + 传统 RAG 解决“能不能答”
pi / Nanobot + 知识图谱 / 语境图谱解决“能不能答对、能不能解释、能不能办理、能不能审计”
```

### 4\.4 推荐演进路线

项目不建议一次性重构全部能力，而应分阶段演进：

| 阶段 | 建设重点 | 目标 |
|---|---|---|
| 阶段 1：保留一期能力 | 保留 Dify RAG 或现有标准 RAG，补齐评测集、召回日志、错误分类和人工标注。 | 先建立可量化的准确率、召回率、幻觉率和人工复核指标。 |
| 阶段 2：建设知识图谱 | 抽取产品、条款、责任、免责、材料、保单、客户等实体关系。 | 解决文档碎片化和多跳关系问题。 |
| 阶段 3：建设语境图谱 | 增加版本、生效期、失效期、来源、权限、地区、渠道、客户身份和任务状态。 | 解决新旧规则混用、相似但不适用、回答不可审计等问题。 |
| 阶段 4：接入智能体底座 | 用 pi / Nanobot 承载 Agent Loop、Skill、工具调用、人工确认和审计事件。 | 从知识问答升级为可办理业务的智能客服。 |
| 阶段 5：平台化复用 | 将 Skill、图谱、工具、评测和审计能力抽象为平台能力。 | 支持多业务线复用，减少重复建设。 |

### 4\.5 风险与边界

新架构有明显优势，但不应低估落地成本：

1. **图谱建模成本**：产品、条款、责任、免责、材料、流程状态等实体关系需要业务和技术共同建模。
2. **语境治理成本**：时间、版本、权限、来源、客户身份和任务状态需要持续维护。
3. **数据质量风险**：如果实体抽取、关系抽取、版本标注或权限继承不准确，图谱会放大错误。
4. **Agent 可控性风险**：pi / Nanobot 原生能力不足以直接满足保险生产要求，必须增加状态机、策略层、工具网关、人工确认和审计。
5. **不适合一刀切替换**：普通 FAQ、低风险说明类问题仍可继续使用标准 RAG，复杂规则和业务办理场景再进入图谱和智能体链路。

### 4\.6 最终建议

建议将下一阶段目标定义为：

```text
建设一套面向保险智能客服的“可控智能体 + 语境图谱 RAG”底座。
```

具体策略如下：

1. **保留一期标准 RAG**，作为文本证据召回和低风险问答基础。
2. **优先建设知识图谱**，围绕保险产品、条款、保单、客户、材料和理赔流程打通实体关系。
3. **逐步建设语境图谱**，重点治理时间、版本、权限、证据、客户身份和任务状态。
4. **采用 Nanobot / pi-mono 作为智能体底座参考**，优先落地 MCP 工具接入、Skill 编排、上下文治理和审计事件。
5. **用评测闭环驱动迭代**，以真实问题集衡量回答准确率、语境命中率、引用正确率、人工复核率和投诉复盘效果。

最终目标不是做一个更复杂的问答系统，而是把智能客服从“文档问答系统”升级为：

```text
可控、可解释、可审计、可办理业务、可持续演进的企业级智能体系统。
```
