## 一、智能体底座能力对比（pi\-mono / Nanobot）\& Skill 运行机制

### 1\.1 pi\-mono 与 Nanobot 简易对比

两款均为自研 / 开源智能体底座，支撑 Skill 插件、多轮决策、任务编排，核心定位与特性对比如下：

|对比项|pi\-mono|Nanobot|
|---|---|---|
|核心定位|轻量化单体底座，侧重**快速部署、极简调度**，面向标准化会话、单任务智能体|模块化分布式底座，侧重**插件生态、复杂任务编排、多 Skill 协同**，面向企业级复杂场景|
|架构模式|单进程为主，组件耦合度中等，启动快、运维简单|微服务拆分设计，底座与插件解耦，支持集群扩容、多实例部署|
|资源开销|低，适合边缘 / 单机部署、轻量业务|中等偏高，适合服务器集群、高并发业务场景|
|扩展能力|支持基础 Skill 加载，复杂多技能串联能力弱|原生支持 Skill 组合、权限隔离、版本管理、动态上下线|
|适配场景|单一场景问答、简单工具调用、对内小型助手|多业务线智能体、复杂流程编排、第三方插件接入|

### 1\.2 统一 Skill 加载机制（两款底座通用逻辑）

Skill 即智能体可调用的**能力插件**（检索、工具调用、接口请求、图谱查询等），加载流程标准化：

1. **配置注册**：Skill 元信息（名称、描述、入参、触发规则、依赖）写入配置文件 / 配置中心；

2. **动态扫描**：底座启动时自动扫描指定目录 / 接口，加载所有已注册 Skill，完成实例初始化；

3. **能力注册**：将 Skill 能力描述注入智能体提示词池，让大模型感知 “当前可调用哪些工具”；

4. **运行调用**：会话过程中，大模型判断需要对应能力 → 底座路由请求至目标 Skill → 执行并返回结果；

5. **热更新**（Nanobot 强化）：支持不停机新增 / 停用 Skill，pi\-mono 需重启生效。

### 1\.3 自主智能体：React 范式 vs 龙虾 \(OpenClaw\) 类 Skill 机制区别

目前项目中两类主流自主决策智能体实现方案，核心是**思考 \- 行动**闭环逻辑差异：

#### 1\. React 范式智能体（通用标准范式）

1. **核心逻辑**：固定循环 `Thought(思考) → Action(行动) → Observation(观察结果)`，轮询执行直至任务结束；

2. **决策特点**：大模型主导全流程决策，每一步都由 LLM 判断**下一步做什么、调用哪个 Skill**，规则弱、灵活性强；

3. **Skill 关联**：Skill 仅作为 “执行工具”，无内置业务逻辑，纯被动接收指令执行；

4. **优缺点**

    - 优势：通用型强，适配各类未知场景，无需硬编码流程；

    - 劣势：步骤不可控、链路长、耗时高，复杂任务易出现决策漂移。

#### 2\. 龙虾 \(OpenClaw\) 类智能体（定制化 Skill 驱动范式）

1. **核心逻辑**：**Skill 分层编排 \+ 预设流程路由**，在 React 基础上增加**规则引擎、子任务分片、状态机管控**；

2. **决策特点**：

    - 粗粒度：由底座 / 状态机预设任务主干流程、Skill 调用顺序；

    - 细粒度：仅在单 Skill 内部交由 LLM 做局部决策，整体链路可控；

3. **Skill 关联**：Skill 不再是纯工具，部分内置**业务规则、上下文缓存、分支判断**，具备半自主能力；

4. **优缺点**

    - 优势：流程可控、执行效率高、出错率低，适合企业固定业务流程；

    - 劣势：新增复杂流程需调整编排规则，极致灵活度弱于原生 React。

#### 核心区别总结

- **React**：LLM 全权决策，**流程自由、无预设链路**；

- **龙虾类**：底座 \+ 状态机控主干，LLM 仅做局部决策，**流程标准化、可控、高效**。

## 二、RAG 检索体系对比：标准 RAG / 知识图谱 RAG / 语境图谱 RAG

### 2\.1 三者核心定义与本质区别

|对比维度|传统标准 RAG（分块文本检索）|知识图谱 RAG（GraphRAG）|语境图谱 RAG（Context Graph RAG）|
|---|---|---|---|
|存储载体|非结构化文本块 \+ 向量库|结构化三元组（实体 \- 关系 \- 实体）\+ 图数据库|三元组 \+ **上下文元数据**（时间 / 版本 / 场景 / 约束）\+ 图数据库|
|核心能力|文本语义相似度匹配，碎片化信息召回|实体关联、多跳推理，梳理知识关系|关系推理 \+ **语境过滤、歧义消除、条件匹配**|
|解决问题|基础文档问答、简单信息查找|实体关联查询、跨内容串联、多跳问题|一词多义、版本差异、时效约束、场景化规则问答|
|数据形态|纯文本 Chunk，无结构化关联|静态事实，无场景 / 时间区分|动态带约束事实，同实体多语境独立存储|

### 2\.2 技术栈 \& 开源属性论证（重点：知识图谱技术归属）

#### 2\.2\.1 标准 RAG

- **核心组件**：文档切片、Embedding 向量模型、向量数据库、BM25 稀疏检索、Rerank 重排模型；

- **技术来源**：**全开源生态**，主流框架：LangChain、LlamaIndex、Dify、Qdrant、Milvus、BGE 系列模型；

- **现状**：项目一期当前采用方案。

#### 2\.2\.2 知识图谱（GraphRAG）

1. **底层图数据库**
Neo4j、NebulaGraph、ArangoDB、JanusGraph 均为**开源产品**，社区版可商用、无厂商绑定；

2. **图谱构建 \& 抽取**
实体 / 关系抽取工具：DeepKE、spaCy、OpenNRE、大模型 Prompt 抽取，**全部开源**；

3. **检索编排框架**
NL2Cypher、GraphQA 能力基于 LangChain、LlamaIndex 开源组件封装，微软 GraphRAG 也为开源项目；

4. **关键结论**
✅ **知识图谱 \& GraphRAG 属于通用开源技术，并非某 IDEA 独有技术**。
任何团队均可基于开源组件独立搭建、二次开发，无技术壁垒与版权限制。

#### 2\.2\.3 语境图谱 RAG（进阶方案）

- **技术本质**：**知识图谱的超集与升级**，在开源知识图谱基础上，扩展「语境属性抽取、条件过滤查询、上下文绑定」能力；

- **底层依赖**：复用开源图数据库、开源编排框架，仅新增**语境解析、条件过滤**自研 / 开源逻辑；

- **技术属性**：底层底座开源，业务层语境规则可根据项目需求自定义开发。

### 2\.3 检索流程简要说明

1. **标准 RAG**：文档分块 → 向量化入库 → 向量 \+ 关键词粗召回 → 重排 → LLM 总结回答；

2. **知识图谱 RAG**：文本分块 \+ 三元组抽取 → 向量库 \+ 图数据库双存储 → 多路召回（文本 \+ 图谱多跳查询）→ 融合重排；

3. **语境图谱 RAG**：三元组挂载语境属性（时间 / 版本 / 场景）→ 解析用户问题约束 → **带条件过滤的图谱检索** \+ 文本检索融合 → 精准匹配后回答。

## 三、新架构 vs 项目一期传统方案：能力差异、提升与优势

### 3\.1 项目一期现状回顾

项目一期采用 **Dify 类传统工作流 \+ 多路召回标准 RAG**：

- 实现方式：文档固定分块、向量 \+ BM25 多路召回、文本片段拼接、大模型统一总结输出；

- 能力边界：仅支持**纯文本语义检索**，无结构化知识、无关系推理、无语境约束能力；

- 现存问题：易出现语义歧义、规则类问答出错、复杂多跳问题回答混乱、大模型幻觉偏高。

### 3\.2 新整体架构（底座 \+ 进阶 RAG）综合能力对比

|能力维度|一期传统 Dify \+ 标准 RAG 方案|新方案（龙虾类自主智能体 \+ 语境图谱 RAG）|
|---|---|---|
|知识形态|碎片化文本，无关联关系|结构化图谱 \+ 文本结合，知识具备完整关联链路|
|复杂推理|不支持多跳推理、实体关联查询|原生支持实体多跳、跨文档知识串联|
|语境 / 规则适配|无法区分时间、版本、场景约束，规则问答易出错|精准识别约束条件，按语境过滤知识，适配保险 / 法务 / 制度类规则|
|智能体调度|固定工作流，灵活性弱，复杂任务易卡顿|龙虾类 Skill 编排 \+ 状态机管控，流程可控、执行效率高，支持多技能协同|
|错误率 / 幻觉|偏高，歧义场景、规则场景需人工复核|大幅降低，知识可溯源、条件强约束，幻觉显著减少|
|扩展能力|仅扩展文档，无结构化能力扩展|支持 Skill 插件热更新、图谱迭代、外部工具接入，生态扩展性强|
|运维 \& 可控性|流程黑盒，问题定位难|Skill、图谱、检索链路分层，日志完整，问题快速定位|

### 3\.3 核心提升与业务优势

#### 1\. 技术层面提升

1. **知识能力升级**：从「文本匹配」升级为**语义理解 \+ 关系推理 \+ 语境感知**，突破传统 RAG 碎片化短板；

2. **架构解耦**：智能体底座与检索能力分层，Skill、图谱、向量库各司其职，便于迭代、扩容、维护；

3. **技术自主可控**：全栈基于开源技术搭建，无闭源组件依赖，可长期深度定制。

#### 2\. 业务层面优势

1. **问答准确率大幅提升**
针对制度、条款、理赔、合规等**带时间 / 版本 / 场景约束的核心业务**，解决一词多义、新旧规则混淆问题，减少人工校验成本；

2. **复杂任务承载能力增强**
支持多技能组合、多子任务拆解、深度调研类问答，不再局限于简单一问一答；

3. **平台复用性更强**
底座 \+ Skill \+ 图谱架构为通用能力，可快速复用到多条业务线，避免重复造轮子；

4. **稳定性与可控性优化**
龙虾类智能体预设主干流程，相比纯自由决策的工作流，线上运行更稳定、异常更少。

#### 3\. 落地价值总结

相较于一期传统分块 RAG 方案，新架构**不仅解决了当前知识库准确率不足、场景适配差的痛点**，同时搭建了一套可长期演进的企业级 AI 智能体 \+ 知识服务底座，既能满足现有业务升级，也支撑后续复杂智能体、深度调研类功能的迭代拓展。

## 四、整体结论

1. **底座层面**：选用龙虾 \(OpenClaw\) 类 Skill 驱动智能体，相比通用 React 范式、轻量化 pi\-mono，更适配企业固定业务流程，**流程可控、执行高效、运维友好**，Nanobot 模块化架构也更利于长期平台化建设；

2. **RAG 层面**：知识图谱、语境图谱均基于成熟开源技术栈实现，无独家技术绑定；相比一期标准 RAG，**结构化知识 \+ 语境过滤**是解决规则类、复杂类问答的核心技术路径；

3. **整体方案**：新架构在**知识理解、推理能力、场景适配、任务复杂度、扩展性**上全面优于一期传统 Dify 工作流 \+ 标准 RAG 方案，可有效降低业务错误率、减少人工成本，同时构建长期技术壁垒，建议作为项目下一阶段核心升级方向。
