## 一、智能体底座能力对比（pi\-mono / Nanobot）\& Skill 运行机制

### 1\.1 pi\-mono\(网易\) 、 Nanobot\(IDEA\)、百度千帆 简易对比

本节针对项目及行业主流三款智能体底座：**pi\-mono、Nanobot、百度千帆**做轻量化简易能力对比。三款底座均支持Skill插件调用\(\)、多轮会话、智能任务调度，但架构定位、可控性、业务适配性差异极大，为后续自主智能体机制选型提供底座层面依据。

|对比项|pi\-mono|Nanobot|百度千帆智能体底座|
|---|---|---|---|
|核心定位|轻量化单体底座，侧重**快速部署、极简调度**，面向标准化单任务会话|模块化分布式底座，侧重**插件生态、复杂任务编排、多Skill协同**，面向企业级复杂场景|通用云上智能体底座，侧重**模型原生自主规划、零编排快速落地**，面向通用轻量化问答场景|
|架构模式|单进程架构，组件耦合中等，启动快、运维极简<br>|微服务解耦设计，底座与插件隔离，支持集群扩容、多实例高可用部署|云上托管黑盒架构，无底层部署能力，仅提供可视化配置与工具调用能力|
|资源开销|极低，适配单机、边缘轻量化部署|中等偏高，适配服务器集群、高并发业务|无需自建资源，云端算力托管，无运维成本但私有化适配差|
|Skill管控能力|仅支持基础Skill加载，不支持复杂编排、优先级管控、热更新|原生支持Skill组合、权限隔离、版本管理、不停机热更新、分层编排|无Skill精细化管控，工具自由调用、无流程约束、无业务规则绑定|
|流程可控性|一般，仅支持简单固定流程，复杂任务易失控|极强，支持状态机管控、SOP固化、流程溯源、异常管控|极弱，纯模型自主决策，执行链路随机、不可控、无法溯源|
|适配场景|内部轻量助手、简单问答、单工具调用场景|企业核心业务、标准化流程、多技能协同、需合规溯源场景|通用创意问答、无固定SOP的轻量化场景、快速演示落地|

**小结**：三款底座定位完全互补，pi\-mono适合轻量化快速落地，百度千帆适合通用无规范场景快速搭建，**Nanobot适配企业标准化、高可控、可迭代的核心业务场景**，也是本次项目架构升级的最优底座选择。

### 1\.2 统一 Skill 加载机制（两款底座通用逻辑）

Skill 即智能体可调用的**能力插件**（检索、工具调用、接口请求、图谱查询等），加载流程标准化：

1. **配置注册**：Skill 元信息（名称、描述、入参、触发规则、依赖）写入配置文件 / 配置中心；

2. **动态扫描**：底座启动时自动扫描指定目录 / 接口，加载所有已注册 Skill，完成实例初始化；

3. **能力注册**：将 Skill 能力描述注入智能体提示词池，让大模型感知 “当前可调用哪些工具”；

4. **运行调用**：会话过程中，大模型判断需要对应能力 → 底座路由请求至目标 Skill → 执行并返回结果；

5. **热更新**（Nanobot 强化）：支持不停机新增 / 停用 Skill，pi\-mono 需重启生效。

### 1.2.1 Nanobot 与 pi-mono Skill 机制差异及生命周期对比

虽然两款底座均支持“在 System Prompt 中注入 Skill 索引，模型按需通过工具读取并执行”的轻量化/渐进式加载模式，但在**发现扫描、读取注入、上下文压缩、历史裁剪/完全消失**这四个生命周期阶段上存在显著差异：

| 生命周期阶段 | Nanobot 机制 | pi-mono 机制 |
| :--- | :--- | :--- |
| **1. 发现与扫描<br>(Discovery & Scanning)** | <ul><li>**严格路径约束**：仅扫描直接子目录下的固定文件名 `SKILL.md`（如 `<workspace>/skills/<name>/SKILL.md`）。</li><li>**覆盖与禁用**：支持 Workspace Skill 同名覆盖（Shadow）内置 Builtin Skill；支持 `disabled_skills` 硬过滤。</li><li>**前置依赖校验**：在 Frontmatter 中声明 `requires.bins/env`。依赖不满足时，会阻止 Full-injection 并标记为 `unavailable`。</li></ul> | <ul><li>**多源灵活扫描**：支持项目级（`.pi/skills/`）、用户级（`~/.pi/agent/skills/`）、Settings 配置及 CLI 参数等多种路径。</li><li>**文件级识别**：除了目录外，支持直接将扫描目录下的**单个 `.md` 文件**识别为 Skill。</li><li>**Frontmatter 校验**：对 `name`（长度/连字符字符集限制）和 `description`（必填）进行严格正则和长度校验，不合规直接拒绝加载。</li></ul> |
| **2. 读取与注入<br>(Reading & Injection)** | <ul><li>**分层注入**：区分 `always`（常驻）与普通 Skill。带有 `always: true` 的 Skill **全文自动注入** System Prompt 的 `# Active Skills` 中。</li><li>**去重设计**：已被全文注入的 Always Skill 会从 `# Skills` 索引摘要中排除，避免重复占用上下文。</li><li>**工具调用**：普通 Skill 注入索引和路径，大模型在推理过程中通过内置 `read_file` 工具读取全文。</li></ul> | <ul><li>**全索引注入**：默认不提供常驻全文的 always 机制，所有启用 Skill 均作为索引列表以 XML 格式（`<available_skills>`）稳定注入 System Prompt。</li><li>**工具调用**：大模型使用内置 `read` 工具读取 `location` 指定的 Markdown 全文。</li><li>**显式参数展开**：如果用户使用 `/skill:name` 命令行指令，底座在发送前会解析该 Skill，并将其全文以 XML 标签包裹形式直接注入第一轮 User Message 中，无需模型二次调用工具。</li></ul> |
| **3. 上下文压缩<br>(Context Compression)** | <ul><li>**微压缩 (Microcompact)**：在 Runner 每次请求 LLM 前对副本 `messages_for_model` 进行细粒度优化。当 compactable tools（含 `read_file`）结果数超过 10 且旧结果长度 >= 500 字符时，会将旧 Skill 的返回内容压缩为一行占位符：`[read_file result omitted from context]`。</li><li>**超大结果预算**：单次 Skill 读取若超出 `maxToolResultChars` 预算限制，会被移至本地磁盘保存，上下文中只保留预览和路径。</li></ul> | <ul><li>**会话全局压缩 (Session Compaction)**：当会话总 Token 超过安全窗口水位线时，触发全局 Compaction。从尾部保留最近约 20000 tokens，将较旧的消息（包括 `read` 读取 Skill 的 Tool Call 和 Tool Result）统一交给大模型总结为一段 `compactionSummary` 消息并替代旧历史。</li><li>**单次工具截断**：`read` 工具执行时自带 2000 行 / 50KB 限制，超出则在输出时直接截断。</li></ul> |
| **4. 裁剪与完全消失<br>(Eviction & Cut-off)** | <ul><li>**上下文裁剪 (Snip)**：若微压缩后仍超预算，通过 `_snip_history` 从头部剔除过旧的非 System 消息，旧的 Skill 占位符会被彻底裁掉。</li><li>**重播窗口裁剪**：`Session.get_history()` 在回放会话时，根据 `max_messages` / `max_tokens` 限制截断旧消息，使其不进入下一轮 LLM 上下文。</li><li>**整理归档 (Consolidation)**：后台定期执行 Consolidation，推进 `last_consolidated` 游标，使旧 Skill 读取消息物理移出 Replay 范畴。</li></ul> | <ul><li>**Compaction 稀释**：被总结在 `compactionSummary` 中的 Skill 语义，会在后续会话持续进行、多次 Compaction 触发后被逐渐稀释甚至完全消失。</li><li>**空闲自动压缩 (Idle Auto-compact)**：会话处于空闲状态时，底座执行物理压缩，**仅保留最近 8 条消息**，多余的旧消息归档并从 Session 历史文件中永久删除。若读取 Skill 的动作较旧，它将从 session 中物理消失。</li><li>**索引级持久**：由于 System Prompt 索引由 Resource Loader 静态生成，即使上下文中的 Skill 全文已被压缩或剔除，模型依然能随时看见索引并重新调用 `read` 恢复全文。</li></ul> |

### 1\.3 自主智能体：百度千帆自主智能体、React范式、龙虾 \(OpenClaw\) 类Skill机制区别

目前行业及项目内存在三类主流自主决策智能体实现方案，分别为**百度千帆原生自主智能体、通用React范式智能体、龙虾\(OpenClaw\)Skill驱动智能体**。三者核心差异集中在「任务调度逻辑、Skill工具管控、流程可控性」，可充分佐证项目选用龙虾类Skill机制的必要性，解决传统自由智能体不可控、无规范、难落地的行业痛点。

#### 1\. 百度千帆自主智能体（行业通用自由规划范式）

1. **核心逻辑**：纯大模型自主规划决策，无预设业务流程、无固定Skill调用链路，依托文心大模型能力自动完成「任务拆解、工具选择、步骤执行」全流程，完全由模型自主判断执行逻辑。

2. **决策特点**：极致自由化，平台仅提供基础工具组件与记忆能力，不干预任务执行路径，模型根据用户问题实时生成执行步骤，支持多场景自主适配、多轮自由对话。

3. **Skill/工具关联**：工具仅为被动执行组件，无分层管控、无业务规则绑定、无调用优先级，模型可随意调用、跳过、重复调用工具，无标准化约束。

4. **优缺点**优势：开箱即用、灵活性极高，适配零散、无固定SOP的通用问答、轻量创意类场景，零编排成本。

5. 劣势：**流程不可控、随机性强、出错率高**，无标准化链路，复杂业务易出现步骤遗漏、工具滥用、决策漂移，无法适配企业标准化业务流程，不支持精准业务管控与问题溯源。

1. **核心逻辑**：固定循环 `Thought(思考) → Action(行动) → Observation(观察结果)`，轮询执行直至任务结束；

2. **决策特点**：大模型主导全流程决策，每一步都由 LLM 判断**下一步做什么、调用哪个 Skill**，规则弱、灵活性强；

3. **Skill 关联**：Skill 仅作为 “执行工具”，无内置业务逻辑，纯被动接收指令执行；

4. **优缺点**

    - 优势：通用型强，适配各类未知场景，无需硬编码流程；

    - 劣势：步骤不可控、链路长、耗时高，复杂任务易出现决策漂移。

#### 2\. React范式智能体（通用标准范式）

1. **核心逻辑**：固定循环 `Thought(思考) → Action(行动) → Observation(观察结果)`，轮询迭代执行直至任务终止，是行业通用的自主智能体基础范式。

2. **决策特点**：全程由LLM主导全流程决策，每一步工具调用、任务推进均由模型自主判断，无人工预设主干流程，规则约束极弱。

3. **Skill 关联**：Skill仅作为纯执行工具，无内置业务逻辑、无流程约束，被动接收模型指令执行，不参与流程管控。

4. **优缺点**优势：通用适配性强，无需硬编码流程，可适配各类未知、非标准化场景。

5. 劣势：执行链路冗长、响应耗时高，复杂业务场景下易出现决策混乱、步骤冗余，线上稳定性差，难以满足企业级业务规范要求。

#### 3\. 龙虾 \(OpenClaw\) 类智能体（项目自研定制Skill驱动范式）

1. **核心逻辑**：**Skill分层编排 \+ 状态机预设流程路由**，在百度千帆、React自由决策的基础上，新增规则引擎、子任务分片、流程管控能力，实现「主干流程可控、局部决策灵活」的平衡架构。

2. **决策特点**：粗粒度管控：由底座状态机预设业务主干SOP、Skill调用顺序、触发条件、优先级，杜绝随机执行；

3. 细粒度灵活适配：单Skill内部交由LLM做局部语义决策、参数适配，保留大模型的智能理解能力。

4. **Skill 关联**：Skill不再是单纯被动工具，可内置**业务规则、上下文缓存、分支判断、权限校验**，具备半自主业务处理能力，贴合企业标准化业务需求。

5. **优缺点**优势：流程标准化、执行高效、出错率极低，支持业务溯源、可控可运维，完美适配保险、合规、制度查询等企业固定业务场景。

6. 劣势：全新未知场景需微调编排规则，极致灵活度略弱于千帆、React自由范式。

1. **核心逻辑**：**Skill 分层编排 \+ 预设流程路由**，在 React 基础上增加**规则引擎、子任务分片、状态机管控**；

2. **决策特点**：

    - 粗粒度：由底座 / 状态机预设任务主干流程、Skill 调用顺序；

    - 细粒度：仅在单 Skill 内部交由 LLM 做局部决策，整体链路可控；

3. **Skill 关联**：Skill 不再是纯工具，部分内置**业务规则、上下文缓存、分支判断**，具备半自主能力；

4. **优缺点**

    - 优势：流程可控、执行效率高、出错率低，适合企业固定业务流程；

    - 劣势：新增复杂流程需调整编排规则，极致灵活度弱于原生 React。

#### 三类智能体核心区别总结（核心立项依据）

- **百度千帆自主智能体**：纯模型自由决策，无流程约束、无Skill管控，适配轻量化通用场景，**企业业务可控性不足**；

- **通用React智能体**：LLM全流程自主迭代，无预设链路、规则薄弱，灵活性强但**复杂业务稳定性差、不可溯源**；

- **龙虾类Skill驱动智能体**：底座控主干、LLM控细节，Skill分层可编排、可固化业务规则，**兼顾智能性与可控性**，是企业级标准化业务落地的最优方案。

- **React**：LLM 全权决策，**流程自由、无预设链路**；

- **龙虾类**：底座 \+ 状态机控主干，LLM 仅做局部决策，**流程标准化、可控、高效**。

## 二、RAG 检索体系对比：标准 RAG / 知识图谱 RAG / 语境图谱 RAG

### 2\.1 三者核心定义与本质区别

|对比维度|传统标准 RAG（分块文本检索）|知识图谱 RAG（GraphRAG）|语境图谱 RAG（Context Graph RAG）|
|---|---|---|---|
|存储载体|非结构化文本块 \+ 向量库|结构化三元组（实体 \- 关系 \- 实体）\+ 图数据库|三元组 \+ **上下文元数据**（时间 / 版本 / 场景 / 约束）\+ 图数据库|
|核心能力|文本语义相似度匹配，碎片化信息召回|实体关联、多跳推理，梳理知识关系|关系推理 \+ **语境过滤、歧义消除、条件匹配**|
|解决问题|基础文档问答、简单信息查找|实体关联查询、跨内容串联、多跳问题|一词多义、版本差异、时效约束、场景化规则问答|
|数据形态|纯文本 Chunk，无结构化关联|静态事实，无场景 / 时间区分|动态带约束事实，同实体多语境独立存储|

### 2\.2 技术栈 \& 开源属性论证（重点：知识图谱技术归属）

#### 2\.2\.1 标准 RAG

- **核心组件**：文档切片、Embedding 向量模型、向量数据库、BM25 稀疏检索、Rerank 重排模型；

- **技术来源**：**全开源生态**，主流框架：LangChain、LlamaIndex、Dify、Qdrant、Milvus、BGE 系列模型；

- **现状**：项目一期当前采用方案。

#### 2\.2\.2 知识图谱（GraphRAG）

1. **底层图数据库**
Neo4j、NebulaGraph、ArangoDB、JanusGraph 均为**开源产品**，社区版可商用、无厂商绑定；

2. **图谱构建 \& 抽取**
实体 / 关系抽取工具：DeepKE、spaCy、OpenNRE、大模型 Prompt 抽取，**全部开源**；

3. **检索编排框架**
NL2Cypher、GraphQA 能力基于 LangChain、LlamaIndex 开源组件封装，微软 GraphRAG 也为开源项目；

4. **关键结论**
✅ **知识图谱 \& GraphRAG 属于通用开源技术，并非某 IDEA 独有技术**。
任何团队均可基于开源组件独立搭建、二次开发，无技术壁垒与版权限制。

#### 2\.2\.3 语境图谱 RAG（进阶方案）

- **技术本质**：**知识图谱的超集与升级**，在开源知识图谱基础上，扩展「语境属性抽取、条件过滤查询、上下文绑定」能力；

- **底层依赖**：复用开源图数据库、开源编排框架，仅新增**语境解析、条件过滤**自研 / 开源逻辑；

- **技术属性**：底层底座开源，业务层语境规则可根据项目需求自定义开发。

### 2\.3 检索流程简要说明

1. **标准 RAG**：文档分块 → 向量化入库 → 向量 \+ 关键词粗召回 → 重排 → LLM 总结回答；

2. **知识图谱 RAG**：文本分块 \+ 三元组抽取 → 向量库 \+ 图数据库双存储 → 多路召回（文本 \+ 图谱多跳查询）→ 融合重排；

3. **语境图谱 RAG**：三元组挂载语境属性（时间 / 版本 / 场景）→ 解析用户问题约束 → **带条件过滤的图谱检索** \+ 文本检索融合 → 精准匹配后回答。

## 三、新架构 vs 项目一期传统方案：能力差异、提升与优势

### 3\.1 项目一期现状回顾

项目一期采用 **Dify 类传统工作流 \+ 多路召回标准 RAG**：
